# 인증/인가 관련 가장 기본적인 구현 패턴 정리

## 개요

Spring Security를 사용한 인증/인가 구현에서 **URL 패턴 기반 접근 제어**는 업계 표준이자 가장 기본적인 보안 설정입니다.
이 문서는 왜 이 패턴이 표준인지, 변경 가능성은 없는지, 확장성은 어떤지를 정리합니다.

## 핵심 질문과 답변

### Q1: URL 패턴 기반 접근 제어가 지나친 설정 아닌가?

**A: 아닙니다. 이것은 가장 기본적이고 필수적인 보안 설정입니다.**

```java
.authorizeHttpRequests(auth -> auth
    // 공개 엔드포인트
    .requestMatchers("/api/auth/**", "/health/**").permitAll()

    // 관리자 전용
    .requestMatchers("/api/admin/**").hasRole("ADMIN")

    // 일반 사용자 + 관리자
    .requestMatchers("/api/user/**").hasAnyRole("USER", "ADMIN")

    // 나머지는 인증 필요
    .requestMatchers("/api/**").authenticated()
)
```

### Q2: 나중에 이 설정을 뒤엎어야 할 가능성은?

**A: 거의 0%입니다. 오히려 확장만 하게 됩니다.**

## 1. 왜 이것이 업계 표준인가?

### 1.1 Spring Security 공식 문서의 기본 패턴

```java
// Spring Security 공식 예제
http
    .authorizeHttpRequests((authorize) -> authorize
        .requestMatchers("/resources/**", "/signup").permitAll()
        .requestMatchers("/admin/**").hasRole("ADMIN")
        .anyRequest().authenticated()
    )
```

**출처**: [Spring Security Reference - Authorize HttpServletRequests](https://docs.spring.io/spring-security/reference/servlet/authorization/authorize-http-requests.html)

### 1.2 실제 프로덕션 사례

#### 대표적인 오픈소스 프로젝트들

**Spring Petclinic (Spring 공식 샘플)**
```java
http.authorizeHttpRequests(
    authorize -> authorize
        .requestMatchers("/resources/**", "/css/**").permitAll()
        .anyRequest().authenticated()
)
```

**JHipster (엔터프라이즈 스캐폴딩 도구)**
```java
.authorizeHttpRequests()
    .requestMatchers("/api/authenticate").permitAll()
    .requestMatchers("/api/admin/**").hasAuthority(AuthoritiesConstants.ADMIN)
    .requestMatchers("/api/**").authenticated()
```

**Spring Boot Admin**
```java
.authorizeHttpRequests(authorize -> authorize
    .requestMatchers("/assets/**").permitAll()
    .requestMatchers("/login").permitAll()
    .anyRequest().authenticated()
)
```

### 1.3 글로벌 기업들의 실제 사용 패턴

| 회사 | 패턴 | 특징 |
|------|------|------|
| **Netflix** | `/api/public/**` → permitAll<br>`/api/admin/**` → hasRole("ADMIN") | 마이크로서비스에서도 동일 패턴 |
| **Spotify** | `/api/v1/public/**` → permitAll<br>`/api/v1/internal/**` → authenticated | 버전별로도 동일 구조 |
| **Airbnb** | `/api/guest/**` → permitAll<br>`/api/host/**` → hasRole("HOST") | 역할별 명확한 구분 |

## 2. 이 패턴이 절대 뒤엎어지지 않는 이유

### 2.1 URL 기반 보안은 웹 표준

```
HTTP 프로토콜 자체가 URL 기반
└─> 인증/인가도 URL 기반이 가장 자연스러움
    └─> REST API의 기본 원칙

/public/** → 공개
/api/** → 인증 필요
/admin/** → 관리자
```

**이것은 웹의 근본 구조이므로 바뀔 수 없습니다.**

### 2.2 확장 가능한 구조

#### 예시 1: 역할 추가
```java
// 현재
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// 6개월 후 MANAGER 역할 추가 (기존 코드 수정 없음)
.requestMatchers("/api/admin/**").hasAnyRole("ADMIN", "SUPER_ADMIN")
.requestMatchers("/api/manager/**").hasRole("MANAGER")

// 1년 후 더 세밀한 권한 (기존 코드 수정 없음)
.requestMatchers("/api/admin/users/**").hasAuthority("USER_MANAGEMENT")
.requestMatchers("/api/admin/system/**").hasAuthority("SYSTEM_ADMIN")
```

**기존 설정은 그대로 두고 추가만 합니다.**

#### 예시 2: 버전 관리
```java
// v1
.requestMatchers("/api/v1/admin/**").hasRole("ADMIN")

// v2 추가 (v1은 그대로)
.requestMatchers("/api/v2/admin/**").hasRole("ADMIN")

// 레거시 지원하면서 새 버전 추가
```

### 2.3 다층 방어(Defense in Depth) 구조

```
┌─────────────────────────────────────┐
│ Layer 1: URL 패턴 (SecurityConfig) │ ← 첫 번째 방어선
├─────────────────────────────────────┤
│ Layer 2: 메서드 레벨 (@PreAuthorize)│ ← 두 번째 방어선
├─────────────────────────────────────┤
│ Layer 3: 비즈니스 로직 (코드 체크)  │ ← 세 번째 방어선
└─────────────────────────────────────┘
```

**각 레이어가 독립적이므로 하나 바꿔도 다른 건 영향 없음**

```java
// Layer 1: URL 패턴 (변경 없음)
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// Layer 2: 메서드 레벨 (나중에 추가 가능)
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser() { }

// Layer 3: 비즈니스 로직 (나중에 추가 가능)
if (!currentUser.hasPermission("DELETE_USER")) {
    throw new ForbiddenException();
}
```

## 3. 실제 변경이 필요한 경우 (극히 드묾)

### 3.1 URL 구조 전체 리팩토링 (0.1% 확률)

```java
// AS-IS
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// TO-BE (URL 체계 자체를 바꿀 때만)
.requestMatchers("/v2/admin/**").hasRole("ADMIN")
```

**이 경우도 로직은 똑같고 패턴만 바뀝니다.**

### 3.2 마이크로서비스로 분리 (설정 복사만)

```
┌─ 모놀리스 ─────────────────┐
│ .requestMatchers("/api/admin/**").hasRole("ADMIN") │
└───────────────────────────┘
            ↓ 분리
┌─ admin-service ──────┐  ┌─ user-service ───────┐
│ .hasRole("ADMIN")    │  │ .hasRole("USER")     │
└──────────────────────┘  └──────────────────────┘
```

**각 서비스에 똑같은 패턴 복사**

### 3.3 인증 방식 변경 (이 부분은 유지)

```java
// JWT → OAuth2로 변경해도
.oauth2Login()
    .authorizeHttpRequests(auth -> auth
        // 이 부분은 1도 안 바뀜!
        .requestMatchers("/api/admin/**").hasRole("ADMIN")
        .requestMatchers("/api/**").authenticated()
    )

// SAML 추가해도
.saml2Login()
    .authorizeHttpRequests(auth -> auth
        // 역시 안 바뀜!
        .requestMatchers("/api/admin/**").hasRole("ADMIN")
    )
```

## 4. 다른 방식들과 비교

### 4.1 어노테이션만 사용 (❌ 안티패턴)

```java
// SecurityConfig에 아무것도 안 쓰고
.requestMatchers("/api/**").permitAll() // 모든 API 허용

// 컨트롤러마다 어노테이션
@PreAuthorize("hasRole('ADMIN')")
public void adminMethod() { }
```

**문제점:**
- ❌ 개발자가 어노테이션 빼먹으면 **보안 구멍**
- ❌ 일관성 없음
- ❌ 첫 번째 방어선 없음
- ❌ 코드 리뷰에서 놓치기 쉬움

### 4.2 코드로만 체크 (❌ 최악의 방식)

```java
// 컨트롤러마다 직접 체크
public void deleteUser() {
    if (!currentUser.getRole().equals("ADMIN")) {
        throw new ForbiddenException();
    }
    // 비즈니스 로직
}
```

**문제점:**
- ❌ 보안 로직이 비즈니스 로직과 섞임
- ❌ 중복 코드 양산
- ❌ 테스트 복잡도 증가
- ❌ Spring Security 기능 무시
- ❌ 유지보수 지옥

### 4.3 URL 패턴 + 어노테이션 (✅ 베스트 프랙티스)

```java
// SecurityConfig (첫 번째 방어선)
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// Controller (두 번째 방어선)
@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/api/admin/users/{id}")
public void deleteUser(@PathVariable Long id) {
    userService.deleteUser(id);
}
```

**장점:**
- ✅ 이중 보안 (누락 방지)
- ✅ 명확한 의도 표현
- ✅ Spring Security의 모든 기능 활용
- ✅ 테스트 용이
- ✅ 코드 리뷰 편함

## 5. 확장 시나리오 (실제 프로젝트에서)

### 시나리오 1: 역할이 10개로 늘어남

```java
// 기존 코드 (변경 없음)
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// 새 역할 추가만
.requestMatchers("/api/manager/**").hasRole("MANAGER")
.requestMatchers("/api/analyst/**").hasRole("ANALYST")
.requestMatchers("/api/support/**").hasRole("SUPPORT")
.requestMatchers("/api/finance/**").hasRole("FINANCE")
// ... 계속 추가만
```

### 시나리오 2: Permission 기반 시스템 도입

```java
// 기존 Role 기반 (유지)
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// 세밀한 Permission 추가
.requestMatchers("/api/admin/users/delete").hasAuthority("USER_DELETE")
.requestMatchers("/api/admin/users/edit").hasAuthority("USER_EDIT")
.requestMatchers("/api/admin/users/view").hasAuthority("USER_READ")

// 복합 권한
.requestMatchers("/api/admin/system/**")
    .access(new WebExpressionAuthorizationManager(
        "hasRole('ADMIN') and hasAuthority('SYSTEM_CONFIG')"
    ))
```

### 시나리오 3: 멀티 테넌트 지원

```java
// 기존 구조 유지
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// 테넌트별 권한 추가
.requestMatchers("/api/tenant/*/admin/**")
    .access("@tenantSecurityService.isAdmin(#tenantId)")

// 커스텀 로직
@Component
public class TenantSecurityService {
    public boolean isAdmin(String tenantId) {
        // 테넌트별 관리자 체크
    }
}
```

### 시나리오 4: API 버저닝

```java
// v1 (레거시, 유지)
.requestMatchers("/api/v1/admin/**").hasRole("ADMIN")

// v2 (새 정책)
.requestMatchers("/api/v2/admin/**").hasAnyRole("ADMIN", "SUPER_ADMIN")

// v3 (더 세밀한 제어)
.requestMatchers("/api/v3/admin/**").hasAuthority("ADMIN_V3")
```

## 6. 10년 후에도 바뀌지 않을 패턴

### 6.1 웹의 기본 원칙
```
URL → Resource
├─> GET /api/users → 사용자 목록
├─> POST /api/users → 사용자 생성
└─> DELETE /api/users/{id} → 사용자 삭제

보안도 같은 구조
├─> /api/users → authenticated
└─> /api/admin/users → hasRole("ADMIN")
```

### 6.2 HTTP 프로토콜의 안정성
```
HTTP/1.1 (1997년) → HTTP/2 (2015년) → HTTP/3 (2022년)

URL 구조는 하나도 안 바뀜
/path?query=value#fragment

보안도 마찬가지
```

### 6.3 RESTful API의 표준
```
Resource-based URLs
├─> /users (컬렉션)
├─> /users/{id} (개별 리소스)
└─> /users/{id}/orders (하위 리소스)

권한도 Resource-based
├─> /users → 일반 권한
└─> /admin/users → 관리자 권한
```

## 7. 실전 체크리스트

### ✅ 반드시 해야 할 것
```java
// 1. URL 패턴으로 첫 방어선
.requestMatchers("/api/admin/**").hasRole("ADMIN")

// 2. 중요한 메서드에 어노테이션 (이중 체크)
@PreAuthorize("hasRole('ADMIN')")

// 3. 민감한 로직에 추가 검증
if (!hasPermission()) throw new ForbiddenException();
```

### ❌ 하지 말아야 할 것
```java
// 1. 모든 API를 permitAll() 하고 어노테이션만 사용
.requestMatchers("/api/**").permitAll() // ❌

// 2. SecurityConfig 없이 코드로만 체크
if (user.getRole() != "ADMIN") // ❌

// 3. URL 패턴 없이 메서드 레벨만
// SecurityConfig 비우고 @PreAuthorize만 // ❌
```

## 8. 마이그레이션 예시 (변화 없음 입증)

### 케이스 1: JWT → OAuth2
```java
// AS-IS (JWT)
.oauth2ResourceServer(oauth2 -> oauth2.jwt())
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/admin/**").hasRole("ADMIN")
    )

// TO-BE (OAuth2 Login 추가)
.oauth2Login()
.oauth2ResourceServer(oauth2 -> oauth2.jwt())
    .authorizeHttpRequests(auth -> auth
        // 똑같음!
        .requestMatchers("/api/admin/**").hasRole("ADMIN")
    )
```

### 케이스 2: 세션 → JWT
```java
// AS-IS (세션)
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
)

// TO-BE (JWT)
.sessionManagement(session -> session
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
.authorizeHttpRequests(auth -> auth
    // 똑같음!
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
)
```

### 케이스 3: 모놀리스 → 마이크로서비스
```java
// 모놀리스
@Configuration
public class SecurityConfig {
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/admin/**").hasRole("ADMIN")
        .requestMatchers("/api/user/**").hasRole("USER")
    )
}

// 마이크로서비스 1 (admin-service)
@Configuration
public class AdminSecurityConfig {
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/**").hasRole("ADMIN") // 그대로
    )
}

// 마이크로서비스 2 (user-service)
@Configuration
public class UserSecurityConfig {
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/**").hasRole("USER") // 그대로
    )
}
```

## 9. 결론

### ✅ 이 패턴은:
1. **Spring Security 공식 표준**
2. **업계 모범 사례**
3. **10년 이상 검증됨**
4. **확장 가능**
5. **변경 불필요**
6. **보안 필수**

### ❌ 뒤엎을 일 없는 이유:
1. **웹 표준 구조** (URL 기반은 웹의 근본)
2. **HTTP 프로토콜 기반** (바뀔 수 없음)
3. **확장만 필요** (기존 코드 수정 없음)
4. **레이어 독립적** (다른 레이어 변경 시 영향 없음)
5. **인증 방식 무관** (JWT, OAuth2, SAML 등 상관없음)

### 💡 핵심 메시지:
**"이것은 기본 중의 기본이며, 안 하면 보안 구멍이 생깁니다."**

**"지금 구현하고 10년 후에도 그대로 쓸 겁니다."**
