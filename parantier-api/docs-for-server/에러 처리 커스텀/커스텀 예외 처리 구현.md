# 전역 예외 처리 (Global Exception Handling) 구현

## 개요

Spring Boot 애플리케이션에서 발생하는 모든 예외를 일관된 형식으로 처리하기 위해
**전역 예외 처리 체계**를 구축했습니다.

## 구성 요소

```
전역 예외 처리 시스템
├── 1. ErrorCode Enum (에러 코드 중앙 관리)
├── 2. Custom Exception Classes (커스텀 예외 클래스)
├── 3. ErrorResponse DTO (통일된 에러 응답 형식)
└── 4. GlobalExceptionHandler (전역 예외 처리기)
```

---

## 1. ErrorCode Enum (에러 코드 중앙 관리)

**파일**: `src/main/java/com/mapo/palantier/common/exception/ErrorCode.java`

### 목적
- 에러 코드를 한 곳에서 중앙 관리
- 매직 스트링 제거 (컴파일 타임 안정성 확보)
- HTTP 상태 코드와 에러 메시지를 함께 관리

### 코드
```java
package com.mapo.palantier.common.exception;

import lombok.Getter;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;

@Getter
@RequiredArgsConstructor
public enum ErrorCode {
    // 인증 관련 (401)
    AUTHENTICATION_FAILED("AUTHENTICATION_FAILED", "인증에 실패했습니다", HttpStatus.UNAUTHORIZED),
    INVALID_CREDENTIALS("INVALID_CREDENTIALS", "이메일 또는 비밀번호가 일치하지 않습니다", HttpStatus.UNAUTHORIZED),

    // 권한 관련 (403)
    ACCOUNT_INACTIVE("ACCOUNT_INACTIVE", "비활성화된 계정입니다", HttpStatus.FORBIDDEN),
    ACCESS_DENIED("ACCESS_DENIED", "접근 권한이 없습니다", HttpStatus.FORBIDDEN),

    // 리소스 관련 (404)
    USER_NOT_FOUND("USER_NOT_FOUND", "사용자를 찾을 수 없습니다", HttpStatus.NOT_FOUND),

    // 중복 관련 (409)
    DUPLICATE_EMAIL("DUPLICATE_EMAIL", "이미 사용 중인 이메일입니다", HttpStatus.CONFLICT),

    // 서버 에러 (500)
    INTERNAL_SERVER_ERROR("INTERNAL_SERVER_ERROR", "서버 내부 오류가 발생했습니다", HttpStatus.INTERNAL_SERVER_ERROR);

    private final String code;
    private final String message;
    private final HttpStatus httpStatus;
}
```

### 장점
```java
// Before: 매직 스트링 (오타 위험)
throw new DuplicateEmailException("DUPLICATE_EMAIL");  // 오타 내도 컴파일 통과

// After: ErrorCode Enum (컴파일 타임 체크)
throw new DuplicateEmailException(ErrorCode.DUPLICATE_EMAIL);  // IDE 자동완성, 오타 시 컴파일 에러
```

## 2. Custom Exception Classes (커스텀 예외 클래스)

### 공통 구조

모든 커스텀 예외는 다음과 같은 구조를 따릅니다:

```java
@Getter
public class XxxException extends RuntimeException {
    private final ErrorCode errorCode;

    public XxxException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }

    public XxxException(ErrorCode errorCode, Throwable cause) {
        super(errorCode.getMessage(), cause);
        this.errorCode = errorCode;
    }
}
```

### 구현된 Exception 목록

#### 2.1 DuplicateEmailException

**파일**: `src/main/java/com/mapo/palantier/common/exception/DuplicateEmailException.java`

**목적**: 이메일 중복 시 발생
**HTTP 상태**: 409 Conflict

```java
@Getter
public class DuplicateEmailException extends RuntimeException {
    private final ErrorCode errorCode;

    public DuplicateEmailException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }

    public DuplicateEmailException(ErrorCode errorCode, Throwable cause) {
        super(errorCode.getMessage(), cause);
        this.errorCode = errorCode;
    }
}
```

**사용 예시**:
```java
if (userRepository.existsByEmail(email)) {
    throw new DuplicateEmailException(ErrorCode.DUPLICATE_EMAIL);
}
```

#### 2.2 AuthenticationException

**파일**: `src/main/java/com/mapo/palantier/common/exception/AuthenticationException.java`

**목적**: 인증 실패 시 발생 (이메일 없음, 비밀번호 불일치)
**HTTP 상태**: 401 Unauthorized

**사용 예시**:
```java
// 이메일로 사용자 조회 실패
User user = userRepository.findByEmail(email)
        .orElseThrow(() -> new AuthenticationException(ErrorCode.INVALID_CREDENTIALS));

// 비밀번호 불일치
if (!passwordEncoder.matches(password, user.getPassword())) {
    throw new AuthenticationException(ErrorCode.INVALID_CREDENTIALS);
}
```

#### 2.3 AccountInactiveException

**파일**: `src/main/java/com/mapo/palantier/common/exception/AccountInactiveException.java`

**목적**: 비활성화된 계정 접근 시 발생
**HTTP 상태**: 403 Forbidden

**사용 예시**:
```java
if (!user.getIsActive()) {
    throw new AccountInactiveException(ErrorCode.ACCOUNT_INACTIVE);
}
```

#### 2.4 UserNotFoundException

**파일**: `src/main/java/com/mapo/palantier/common/exception/UserNotFoundException.java`

**목적**: 사용자를 찾을 수 없을 때 발생
**HTTP 상태**: 404 Not Found

**사용 예시**:
```java
return userRepository.findByEmail(email)
        .orElseThrow(() -> new UserNotFoundException(ErrorCode.USER_NOT_FOUND));
```

## 3. ErrorResponse DTO (통일된 에러 응답 형식)

**파일**: `src/main/java/com/mapo/palantier/common/dto/ErrorResponse.java`

### 목적
- 모든 에러를 일관된 JSON 형식으로 클라이언트에 반환
- 프론트엔드에서 에러 처리를 예측 가능하게 만듦

### 코드
```java
package com.mapo.palantier.common.dto;

import lombok.AllArgsConstructor;
import lombok.Builder;
import lombok.Getter;
import java.time.LocalDateTime;

@Getter
@Builder
@AllArgsConstructor
public class ErrorResponse {
    private String errorCode;      // 에러 코드 (예: "DUPLICATE_EMAIL")
    private String message;         // 에러 메시지 (예: "이미 사용 중인 이메일입니다")
    private LocalDateTime timestamp; // 에러 발생 시각
    private String path;            // 에러가 발생한 API 경로

    // 정적 팩토리 메서드
    public static ErrorResponse of(String errorCode, String message, String path) {
        return ErrorResponse.builder()
                .errorCode(errorCode)
                .message(message)
                .timestamp(LocalDateTime.now())
                .path(path)
                .build();
    }
}
```

### 응답 예시
```json
{
  "errorCode": "DUPLICATE_EMAIL",
  "message": "이미 사용 중인 이메일입니다",
  "timestamp": "2026-03-22T14:30:00",
  "path": "/api/auth/signup"
}
```

## 4. GlobalExceptionHandler (전역 예외 처리기)

**파일**: `src/main/java/com/mapo/palantier/common/exception/GlobalExceptionHandler.java`

### 목적
- 애플리케이션 전역에서 발생하는 예외를 한 곳에서 처리
- 각 예외 타입별로 적절한 HTTP 상태 코드와 에러 응답 반환

### 핵심 어노테이션

```java
@Slf4j                    // 로깅
@RestControllerAdvice     // 전역 예외 처리 활성화
public class GlobalExceptionHandler {
    // ...
}
```

**@RestControllerAdvice의 역할**:
- Spring이 자동으로 이 클래스를 찾아서 예외 처리기로 등록
- 모든 @RestController에서 발생하는 예외를 이 클래스에서 처리

### 주요 핸들러 메서드

```java
@ExceptionHandler(DuplicateEmailException.class)
public ResponseEntity<ErrorResponse> handleDuplicateEmailException(
        DuplicateEmailException e,
        HttpServletRequest request
) {
    log.warn("Duplicate email: {} - Path: {}", e.getMessage(), request.getRequestURI());

    ErrorResponse errorResponse = ErrorResponse.of(
            e.getErrorCode().getCode(),
            e.getMessage(),
            request.getRequestURI()
    );

    return ResponseEntity
            .status(e.getErrorCode().getHttpStatus())
            .body(errorResponse);
}
```

## 실제 사용 예시 (AuthService)

**파일**: `src/main/java/com/mapo/palantier/user/application/AuthService.java`

### 회원가입 (signup)

```java
@Transactional
public User signup(String email, String password, String username) {
    // 1. 이메일 중복 체크
    if (userRepository.existsByEmail(email)) {
        throw new DuplicateEmailException(ErrorCode.DUPLICATE_EMAIL);
    }

    // 2. 비밀번호 암호화
    String encodedPassword = passwordEncoder.encode(password);

    // 3. User 도메인 객체 생성
    User newUser = User.createNewUser(email, encodedPassword, username);

    // 4. 저장
    userRepository.save(newUser);

    return newUser;
}
```

### 로그인 (login)

```java
@Transactional(readOnly = true)
public String login(String email, String password) {
    // 1. 이메일로 사용자 조회
    User user = userRepository.findByEmail(email)
            .orElseThrow(() -> new AuthenticationException(ErrorCode.INVALID_CREDENTIALS));

    // 2. 비밀번호 검증
    if (!passwordEncoder.matches(password, user.getPassword())) {
        throw new AuthenticationException(ErrorCode.INVALID_CREDENTIALS);
    }

    // 3. 계정 활성화 여부 확인
    if (!user.getIsActive()) {
        throw new AccountInactiveException(ErrorCode.ACCOUNT_INACTIVE);
    }

    // 4. JWT 토큰 생성
    return jwtTokenProvider.generateToken(user.getEmail(), user.getRole().name());
}
```

| Exception                    | HTTP Status | 상태 코드 | 설명                          |

## 전역 예외 처리의 장점

### 1. 일관성 (Consistency)
- 모든 에러가 동일한 JSON 형식으로 응답됨
- 프론트엔드에서 에러 처리 로직을 통일할 수 있음

### 2. 중앙화 (Centralization)
- 에러 처리 로직이 한 곳(GlobalExceptionHandler)에 모여 있음
- 에러 응답 형식 변경 시 한 곳만 수정하면 됨

### 3. 유지보수성 (Maintainability)
- 새로운 예외 추가 시 @ExceptionHandler 메서드만 추가
- ErrorCode Enum으로 에러 코드 중앙 관리

### 4. 타입 안정성 (Type Safety)
- ErrorCode Enum 사용으로 컴파일 타임에 오타 검증
- IDE 자동완성 지원

### 5. 로깅 (Logging)
- 모든 예외가 자동으로 로그에 기록됨
- 디버깅과 모니터링이 용이

## Before vs After 비교

### Before (예외 처리 없이)

```java
// Controller
@PostMapping("/signup")
public ResponseEntity<?> signup(@RequestBody SignupRequest request) {
    try {
        User user = authService.signup(request.getEmail(), request.getPassword(), request.getUsername());
        return ResponseEntity.ok(user);
    } catch (IllegalArgumentException e) {
        return ResponseEntity.badRequest().body(e.getMessage());  // 일관성 없는 응답
    } catch (IllegalStateException e) {
        return ResponseEntity.status(409).body(e.getMessage());   // 매직 넘버
    }
}

// Service
if (userRepository.existsByEmail(email)) {
    throw new IllegalStateException("이미 존재하는 이메일입니다");  // 범용 예외 사용
}
```

**문제점**:
- Controller마다 try-catch 중복
- 에러 응답 형식이 각각 다름
- HTTP 상태 코드 하드코딩 (409)
- 범용 예외(IllegalStateException) 사용으로 의미 불명확

### After (전역 예외 처리 적용)

```java
// Controller (간결해짐!)
@PostMapping("/signup")
public ResponseEntity<User> signup(@RequestBody SignupRequest request) {
    User user = authService.signup(request.getEmail(), request.getPassword(), request.getUsername());
    return ResponseEntity.ok(user);
    // 예외는 GlobalExceptionHandler가 자동 처리
}

// Service (의미 명확!)
if (userRepository.existsByEmail(email)) {
    throw new DuplicateEmailException(ErrorCode.DUPLICATE_EMAIL);  // 명확한 예외 타입
}
```

**개선점**:
- Controller에서 try-catch 제거
- 에러 응답 형식 통일 (ErrorResponse)
- HTTP 상태 코드 자동 매핑
- 의미 있는 커스텀 예외 사용
- ErrorCode Enum으로 타입 안정성 확보

## 결론

**전역 예외 처리 시스템**을 구축하여:

1. 일관성: 모든 에러가 동일한 형식(ErrorResponse)으로 응답
2. 명확성: 각 예외가 의미 있는 이름과 HTTP 상태 코드를 가짐
3. 안정성: ErrorCode Enum으로 컴파일 타임 오류 검증
4. 유지보수성: 중앙 집중식 관리로 변경 시 한 곳만 수정
5. 협업 효율성: 프론트엔드와 명확한 에러 규격 공유

**실무 수준의 견고한 에러 처리 체계**를 완성했습니다!
