# 리프레시 토큰 미구현 현황

## 현재 JWT 인증 구현 상태

### 구현된 기능 ✅

#### 1. 액세스 토큰만 사용하는 기본 JWT 인증
**파일**: `JwtTokenProvider.java`

```java
@Component
public class JwtTokenProvider {
    private final SecretKey secretKey;
    private final long expirationTime;  // Access Token 만료 시간

    /**
     * JWT 액세스 토큰 생성
     */
    public String generateToken(String email, String role) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + expirationTime);

        return Jwts.builder()
                .subject(email)
                .claim("role", role)
                .issuedAt(now)
                .expiration(expiryDate)
                .signWith(secretKey)
                .compact();
    }

    // 토큰 검증, 이메일/역할 추출 메서드들...
}
```

**특징**:
- **액세스 토큰만 생성**
- `application.yml`에서 만료 시간 설정 (예: 1시간)
- 로그인 시 토큰 1개만 반환

#### 2. 로그인 시 액세스 토큰 반환
**파일**: `AuthService.java`

```java
@Transactional(readOnly = true)
public String login(String email, String password) {
    // 1. 사용자 조회 및 검증

            .orElseThrow(() -> new AuthenticationException(ErrorCode.INVALID_CREDENTIALS));

    // 2. 비밀번호 검증

        throw new AuthenticationException(ErrorCode.INVALID_CREDENTIALS);
    }

    // 3. 계정 활성화 확인

        throw new AccountInactiveException(ErrorCode.ACCOUNT_INACTIVE);

    // 4. JWT 액세스 토큰만 생성하여 반환
    return jwtTokenProvider.generateToken(user.getEmail(), user.getRole().name());
}
```

## 미구현된 기능 ❌

### 1. 리프레시 토큰 (Refresh Token)

#### 리프레시 토큰이란?

**Access Token (액세스 토큰)**:
- 짧은 만료 시간 (예: 15분 ~ 1시간)
- API 요청 시 사용
- 탈취 위험 최소화

**Refresh Token (리프레시 토큰)**:
- 긴 만료 시간 (예: 7일 ~ 30일)
- 액세스 토큰 재발급 시에만 사용
- DB에 저장하여 관리 (탈취 시 무효화 가능)

#### 리프레시 토큰 동작 흐름

```
1. 로그인 성공
   ↓
   서버: 액세스 토큰 (1시간) + 리프레시 토큰 (7일) 발급
   ↓
   클라이언트: 두 토큰 모두 저장

2. API 요청
   ↓
   클라이언트: 액세스 토큰과 함께 요청
   ↓
   서버: 액세스 토큰 검증 후 응답

3. 액세스 토큰 만료 (1시간 후)
   ↓
   클라이언트: API 요청 → 401 Unauthorized 응답
   ↓
   클라이언트: 리프레시 토큰으로 재발급 요청
   ↓
   서버: 리프레시 토큰 검증 (DB 조회)
   ↓
   서버: 새 액세스 토큰 발급 (+ 선택적으로 새 리프레시 토큰)
   ↓
   클라이언트: 새 액세스 토큰으로 다시 API 요청

4. 리프레시 토큰도 만료 (7일 후)
   ↓
   클라이언트: 재로그인 필요
```

#### 현재 구현 vs 리프레시 토큰 구현

| 항목 | 현재 구현 (액세스 토큰만) | 리프레시 토큰 구현 시 |
|------|---------------------------|---------------------|
| **로그인 응답** | `{ "token": "..." }` | `{ "accessToken": "...", "refreshToken": "..." }` |
| **토큰 만료 시** | 재로그인 필요 | 리프레시 토큰으로 재발급 |
| **보안성** | 낮음 (긴 만료 시간 필요) | 높음 (짧은 액세스 토큰) |
| **사용자 경험** | 자주 로그아웃됨 | 긴 로그인 유지 |
| **DB 저장** | 불필요 | 리프레시 토큰 저장 필요 |
| **토큰 무효화** | 불가능 (만료 시까지 유효) | 가능 (DB에서 삭제) |

### 4. JwtTokenProvider 수정

```java
@Component
public class JwtTokenProvider {
    private final SecretKey secretKey;
    private final long accessTokenExpiration;   // 액세스 토큰 만료 (예: 1시간)
    private final long refreshTokenExpiration;  // 리프레시 토큰 만료 (예: 7일)

    /**
     * 액세스 토큰 생성
     */
    public String generateAccessToken(String email, String role) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + accessTokenExpiration);

        return Jwts.builder()
                .subject(email)
                .claim("role", role)
                .claim("type", "access")  // 토큰 타입 구분
                .issuedAt(now)
                .expiration(expiryDate)
                .signWith(secretKey)
                .compact();
    }

    /**
     * 리프레시 토큰 생성
     */
    public String generateRefreshToken(String email) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + refreshTokenExpiration);

        return Jwts.builder()
                .subject(email)
                .claim("type", "refresh")  // 토큰 타입 구분
                .issuedAt(now)
                .expiration(expiryDate)
                .signWith(secretKey)
                .compact();
    }

    // 기타 검증 메서드들...
}
```

### 5. AuthService 수정

```java
@Service
@RequiredArgsConstructor
public class AuthService {
    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final JwtTokenProvider jwtTokenProvider;
    private final RefreshTokenRepository refreshTokenRepository;  // 추가

    @Transactional
    public TokenResponse login(String email, String password) {
        // 1. 사용자 조회 및 검증
        User user = userRepository.findByEmail(email)
                .orElseThrow(() -> new AuthenticationException(ErrorCode.INVALID_CREDENTIALS));

        // 2. 비밀번호 검증
        if (!passwordEncoder.matches(password, user.getPassword())) {
            throw new AuthenticationException(ErrorCode.INVALID_CREDENTIALS);
        }

        // 3. 계정 활성화 확인
        if (!user.getIsActive()) {
            throw new AccountInactiveException(ErrorCode.ACCOUNT_INACTIVE);
        }

        // 4. 액세스 토큰 + 리프레시 토큰 생성
        String accessToken = jwtTokenProvider.generateAccessToken(user.getEmail(), user.getRole().name());
        String refreshToken = jwtTokenProvider.generateRefreshToken(user.getEmail());

        // 5. 리프레시 토큰 DB에 저장 (기존 토큰 있으면 삭제)
        refreshTokenRepository.deleteByUserId(user.getId());
        RefreshToken refreshTokenEntity = RefreshToken.create(user, refreshToken, 7);  // 7일
        refreshTokenRepository.save(refreshTokenEntity);

        // 6. 두 토큰 모두 반환
        return TokenResponse.of(accessToken, refreshToken);
    }

    @Transactional
    public String refreshAccessToken(String refreshToken) {
        // 1. 리프레시 토큰 검증 (JWT 서명)
        if (!jwtTokenProvider.validateToken(refreshToken)) {
            throw new AuthenticationException(ErrorCode.INVALID_TOKEN);
        }

        // 2. DB에서 리프레시 토큰 조회
        RefreshToken tokenEntity = refreshTokenRepository.findByToken(refreshToken)
                .orElseThrow(() -> new AuthenticationException(ErrorCode.INVALID_TOKEN));

        // 3. 만료 여부 확인
        if (tokenEntity.isExpired()) {
            refreshTokenRepository.delete(tokenEntity);
            throw new AuthenticationException(ErrorCode.EXPIRED_REFRESH_TOKEN);
        }

        // 4. 사용자 조회
        User user = tokenEntity.getUser();

        // 5. 새 액세스 토큰 발급
        return jwtTokenProvider.generateAccessToken(user.getEmail(), user.getRole().name());
    }

    @Transactional
    public void logout(Long userId) {
        // 리프레시 토큰 삭제 (로그아웃 처리)
        refreshTokenRepository.deleteByUserId(userId);
    }
}
```

### 6. Controller 추가

```java
@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
public class AuthController {
    private final AuthService authService;

    /**
     * 로그인 (액세스 토큰 + 리프레시 토큰 발급)
     */
    @PostMapping("/login")
    public ResponseEntity<TokenResponse> login(@RequestBody LoginRequest request) {
        TokenResponse tokens = authService.login(request.getEmail(), request.getPassword());
        return ResponseEntity.ok(tokens);
    }

    /**
     * 액세스 토큰 재발급
     */
    @PostMapping("/refresh")
    public ResponseEntity<AccessTokenResponse> refresh(@RequestBody RefreshTokenRequest request) {
        String newAccessToken = authService.refreshAccessToken(request.getRefreshToken());
        return ResponseEntity.ok(AccessTokenResponse.of(newAccessToken));
    }

    /**
     * 로그아웃 (리프레시 토큰 무효화)
     */
    @PostMapping("/logout")
    public ResponseEntity<Void> logout(@AuthenticationPrincipal CustomUserDetails userDetails) {
        authService.logout(userDetails.getUserId());
        return ResponseEntity.ok().build();
    }
}
```

## 리프레시 토큰 구현의 장점

### 1. 보안성 향상
- **액세스 토큰 만료 시간 단축** (15분 ~ 1시간)
- 탈취되어도 짧은 시간만 유효
- 리프레시 토큰은 DB에 저장하여 무효화 가능

### 2. 사용자 경험 개선
- 자주 로그인할 필요 없음
- 액세스 토큰 만료 시 자동 재발급
- 긴 로그인 유지 (예: 7일)

### 3. 토큰 무효화 가능
```java
// 로그아웃 시
authService.logout(userId);  // DB에서 리프레시 토큰 삭제

// 강제 로그아웃 (예: 비밀번호 변경 시)
refreshTokenRepository.deleteByUserId(userId);
```

### 4. 보안 이벤트 대응
```java
// 의심스러운 활동 감지 시 모든 토큰 무효화
refreshTokenRepository.deleteByUserId(userId);
// → 모든 기기에서 로그아웃 효과
```

## 현재 구현의 한계

### 1. 액세스 토큰 만료 시 재로그인 필요
```
사용자: API 요청
  ↓
서버: 401 Unauthorized (토큰 만료)
  ↓
사용자: 다시 로그인해야 함 (불편!)
```

### 2. 토큰 무효화 불가능
- 한번 발급된 토큰은 만료 시까지 유효
- 탈취되어도 막을 방법 없음
- 로그아웃해도 토큰은 여전히 유효 (보안 위험!)

### 3. 긴 만료 시간 vs 보안 트레이드오프
- **짧은 만료 시간** (예: 15분) → 보안 좋음, 사용자 경험 나쁨
- **긴 만료 시간** (예: 7일) → 사용자 경험 좋음, 보안 위험

## 결론

### 현재 상태
- ✅ 기본 JWT 인증 구현 완료
- ✅ 액세스 토큰만 사용
- ❌ 리프레시 토큰 미구현

### 추후 구현 필요 사항
1. **refresh_tokens 테이블 생성**
2. **RefreshToken 엔티티 및 Repository 추가**
3. **JwtTokenProvider 확장** (액세스/리프레시 토큰 분리)
4. **AuthService 수정** (로그인 시 두 토큰 발급, 재발급 API 추가)
5. **Controller 추가** (`/refresh`, `/logout` 엔드포인트)
6. **ErrorCode 추가** (토큰 관련 에러 코드)

### 구현 우선순위
- **현재**: 기본 인증 동작 확인 및 테스트
- **다음 단계**: 리프레시 토큰 구현으로 보안성 + 사용자 경험 개선

**리프레시 토큰은 실무에서 거의 필수적으로 사용되는 패턴입니다!**
