# 폼 검증 타이밍 UX 개선

## 문제 상황

### 사용자 피드백
> "아무것도 입력 안했는데 빨간 표시가 막 나와 저장 하면 그때 체크해서 보여주는게 낫지 않냐?"

### 기존 동작
- 폼을 열자마자 또는 필드를 건드리자마자 에러 메시지 표시
- 사용자가 아직 입력하지 않았는데 에러가 보여서 혼란스러움

```tsx
// 기존 코드 - onChange 검증
const form = useForm<SignupFormData>({
  defaultValues: { email: '', username: '', password: '' },
  validatorAdapter: zodValidator(),
  validators: {
    onChange: signupSchema,  // ❌ 입력할 때마다 검증
  },
  onSubmit: async ({ value }) => {
    await authApi.signup(value)
  },
})

// 기존 에러 표시 조건
{field.state.meta.isTouched && field.state.meta.errors.length > 0 && (
  <p className="text-sm text-destructive">
    {String(field.state.meta.errors[0])}
  </p>
)}
```

## 해결 방법

### 1단계: 검증 시점 변경
```tsx
const form = useForm<SignupFormData>({
  defaultValues: {
    email: '',
    username: '',
    password: '',
    passwordConfirm: '',
  },
  validatorAdapter: zodValidator(),
  // validators.onChange 제거 - 실시간 검증 비활성화
  onSubmit: async ({ value }) => {
    // ✅ 제출 시점에 검증
    const result = signupSchema.safeParse(value)
    if (!result.success) {
      // 에러가 있으면 각 필드에 설정
      result.error.errors.forEach((err) => {
        const fieldName = err.path[0] as keyof SignupFormData
        form.setFieldMeta(fieldName, (prev) => ({
          ...prev,
          errorMap: { onSubmit: err.message },
        }))
      })
      return
    }

    try {
      const { passwordConfirm, ...signupData } = value
      await authApi.signup(signupData)
      setSuccess(true)
    } catch (err: any) {
      form.setErrorMap({
        onSubmit: err.response?.data?.message || '회원가입에 실패했습니다.',
      })
    }
  },
})
```

### 2단계: 에러 표시 조건 변경
```tsx
// ✅ 제출 후에만 에러 표시
{field.state.meta.errors.length > 0 && field.state.meta.errorMap?.onSubmit ? (
  <p className="text-sm text-destructive">
    {String(field.state.meta.errors[0])}
  </p>
) : (
  <p className="text-xs text-muted-foreground">
    이메일 형식으로 입력해주세요
  </p>
)}
```

## UX 개선 포인트

### Before (실시간 검증)
```
사용자 행동: [이메일 필드 클릭]
화면:        ❌ "이메일을 입력하세요" (빨간색)

사용자 행동: [이메일 1글자 입력: "a"]
화면:        ❌ "유효한 이메일 형식이 아닙니다" (빨간색)

사용자 행동: [이메일 입력 중: "abc"]
화면:        ❌ "유효한 이메일 형식이 아닙니다" (빨간색)
```
- 사용자가 입력하는 동안 계속 에러 표시
- 심리적 압박감

### After (제출 시 검증)
```
사용자 행동: [이메일 필드 클릭]
화면:        💡 "이메일 형식으로 입력해주세요" (회색 - 도움말)

사용자 행동: [이메일 입력 중: "abc"]
화면:        💡 "이메일 형식으로 입력해주세요" (회색 - 도움말)

사용자 행동: [회원가입 버튼 클릭]
화면:        ❌ "유효한 이메일 형식이 아닙니다" (빨간색)
              (이때 처음으로 에러 표시)
```
- 입력 중에는 도움말만 표시
- 제출 버튼 클릭 후에만 에러 표시
- 사용자 친화적

## 헬퍼 텍스트 추가

### 사용자 피드백
> "헬프 텍스트는 평상시에도 그 밑에 작게 보여주는것도 괜찮을거 같기도"

### 구현
```tsx
{field.state.meta.errors.length > 0 && field.state.meta.errorMap?.onSubmit ? (
  // 에러가 있고 제출한 경우: 에러 메시지 (빨간색)
  <p className="text-sm text-destructive">
    {String(field.state.meta.errors[0])}
  </p>
) : (
  // 기본 상태: 헬퍼 텍스트 (회색)
  <p className="text-xs text-muted-foreground">
    이메일 형식으로 입력해주세요
  </p>
)}
```

### 모든 필드의 헬퍼 텍스트
```tsx
// 이메일
"이메일 형식으로 입력해주세요"

// 사용자 이름
"2자 이상 50자 이하로 입력해주세요"

// 비밀번호
"8자 이상, 대/소문자, 숫자, 특수문자 포함"

// 비밀번호 확인
"비밀번호를 다시 입력해주세요"
```

## 장점

### 1. 사용자 경험 향상
- ✅ 입력 중 방해받지 않음
- ✅ 친절한 가이드 제공 (헬퍼 텍스트)
- ✅ 필요한 시점에만 에러 표시

### 2. 심리적 부담 감소
- ✅ "이미 틀렸다"는 압박감 없음
- ✅ 차근차근 입력할 수 있는 여유

### 3. 명확한 피드백
- ✅ 제출 시점에 모든 에러를 한 번에 확인
- ✅ 무엇을 고쳐야 하는지 명확

## 검증 전략 비교

| 검증 시점 | 장점 | 단점 | 적합한 경우 |
|---------|------|------|-----------|
| **onChange** | 즉각적인 피드백 | 입력 중 방해, 스트레스 | 실시간 데이터 검증 필요 (재고 확인 등) |
| **onBlur** | 필드 이탈 시 검증 | 여전히 입력 중 방해 가능 | 필드별 독립 검증 |
| **onSubmit** | 입력 완료 후 검증 | 에러 발견이 늦음 | 회원가입, 설문조사 등 |
| **Hybrid** | 최적의 UX | 구현 복잡도 증가 | 복잡한 폼 |

## TanStack Form의 검증 모드

```typescript
// 1. onChange 검증
validators: {
  onChange: signupSchema,
}

// 2. onBlur 검증
validators: {
  onBlur: signupSchema,
}

// 3. onSubmit 검증 (수동)
onSubmit: async ({ value }) => {
  const result = signupSchema.safeParse(value)
  if (!result.success) {
    // 에러 처리
  }
}
```
