# 버튼 비활성화 문제 해결

## 문제 상황

### 사용자 피드백
> "중복 확인 안눌러지고 저장도 안눌러져 이거 머가 문제야?"

### 증상
- 이메일 중복 확인 버튼이 클릭되지 않음
- 회원가입 버튼도 클릭되지 않음
- 모든 버튼이 비활성화 상태

## 원인 분석

### 문제가 된 코드
```tsx
// 이메일 중복 확인 버튼
<Button
  type="button"
  variant="outline"
  onClick={handleEmailCheck}
  disabled={
    emailCheckStatus === 'checking' ||
    !field.state.value ||
    field.state.meta.errors.length > 0  // ❌ 문제의 원인
  }
  className="w-24"
>
  {emailCheckStatus === 'checking' ? '확인 중...' : '중복 확인'}
</Button>
```

### 왜 문제가 발생했나?

#### 1. 검증 모드 변경의 부작용
```typescript
// BEFORE: onChange 검증
validators: {
  onChange: signupSchema,  // 입력할 때마다 검증
}

// AFTER: onSubmit 검증
onSubmit: async ({ value }) => {
  const result = signupSchema.safeParse(value)
  // 제출할 때만 검증
}
```

#### 2. errors 배열의 상태
- **onChange 검증**: `field.state.meta.errors`가 실시간으로 업데이트됨
- **onSubmit 검증**: `field.state.meta.errors`가 비어있을 수 있음
- 하지만 검증 로직은 제거했는데, **버튼 disabled 조건은 여전히 errors를 체크**

#### 3. 버튼이 비활성화되는 이유
```tsx
disabled={
  emailCheckStatus === 'checking' ||  // ✅ 정상
  !field.state.value ||                // ✅ 정상
  field.state.meta.errors.length > 0   // ❌ 항상 false가 되어야 하는데...
}
```

실제로는 TanStack Form의 내부 검증 메커니즘 때문에 `errors` 배열이 예상과 다르게 동작했을 가능성

## 해결 방법

### 수정된 코드
```tsx
// ✅ 이메일 중복 확인 버튼
<Button
  type="button"
  variant="outline"
  onClick={handleEmailCheck}
  disabled={
    emailCheckStatus === 'checking' ||
    !field.state.value
    // field.state.meta.errors.length > 0 제거
  }
  className="w-24"
>
  {emailCheckStatus === 'checking' ? '확인 중...' : '중복 확인'}
</Button>
```

### handleEmailCheck 함수도 수정
```tsx
// BEFORE
const handleEmailCheck = async () => {
  const email = form.getFieldValue('email')
  if (!email || form.getFieldMeta('email')?.errors.length > 0) {
    return
  }
  // ...
}

// AFTER
const handleEmailCheck = async () => {
  const email = form.getFieldValue('email')
  if (!email) {  // ✅ errors 체크 제거
    return
  }

  setEmailCheckStatus('checking')
  try {
    const isDuplicate = await authApi.checkEmailDuplicate(email)
    setEmailCheckStatus(isDuplicate ? 'duplicate' : 'available')
  } catch (err) {
    setEmailCheckStatus('idle')
  }
}
```

### 회원가입 버튼 (변경 없음)
```tsx
<form.Subscribe selector={(state) => [state.canSubmit, state.isSubmitting]}>
  {([canSubmit, isSubmitting]) => (
    <Button
      type="submit"
      disabled={
        !canSubmit ||
        isSubmitting ||
        emailCheckStatus !== 'available'
      }
      className="w-full"
    >
      {isSubmitting ? '가입 중...' : '회원가입'}
    </Button>
  )}
</form.Subscribe>
```

## 핵심 원리

### onSubmit 검증 방식의 특징
1. **폼 제출 전**: `field.state.meta.errors`가 비어있거나 예측 불가
2. **폼 제출 후**: `setFieldMeta()`로 수동으로 에러 설정
3. **버튼 disabled 로직**: errors에 의존하면 안 됨

### 올바른 버튼 비활성화 조건

#### 중복 확인 버튼
```tsx
disabled={
  emailCheckStatus === 'checking' ||  // 확인 중일 때
  !field.state.value                   // 이메일이 비어있을 때
}
```
- ✅ 이메일 형식 검증은 하지 않음 (어차피 제출 시 검증됨)
- ✅ 비어있지 않으면 클릭 가능
- ✅ 사용자가 잘못된 형식을 입력해도 중복 확인 가능

#### 회원가입 버튼
```tsx
disabled={
  !canSubmit ||                        // 폼 제출 가능 상태가 아닐 때
  isSubmitting ||                      // 제출 중일 때
  emailCheckStatus !== 'available'     // 이메일 중복 확인이 안 됐을 때
}
```
- ✅ `canSubmit`은 TanStack Form이 관리
- ✅ 이메일 중복 확인 필수 조건 유지

## 검증 전략별 버튼 제어

### onChange/onBlur 검증
```tsx
// 실시간으로 errors가 업데이트되므로 errors 체크 가능
disabled={
  field.state.meta.errors.length > 0 ||
  !field.state.value
}
```

### onSubmit 검증
```tsx
// errors가 실시간 업데이트 안 되므로 errors 체크하면 안 됨
disabled={
  !field.state.value  // 값이 있는지만 체크
}
```

## TanStack Form의 canSubmit

### canSubmit이 false가 되는 경우
1. 폼이 제출 중일 때 (`isSubmitting`)
2. 검증 실패 시 (validators가 설정된 경우)
3. 필수 필드가 비어있을 때

### 우리의 경우
```typescript
const form = useForm<SignupFormData>({
  defaultValues: {
    email: '',
    username: '',
    password: '',
    passwordConfirm: '',
  },
  validatorAdapter: zodValidator(),
  // validators 설정 없음 → canSubmit이 거의 항상 true
  onSubmit: async ({ value }) => {
    // 수동 검증
  },
})
```

- `validators`를 설정하지 않았으므로 `canSubmit`은 기본적으로 true
- 이메일 중복 확인 상태를 별도로 체크해야 함

## 배운 점

### 1. 검증 모드와 UI 로직의 일관성
- 검증 모드를 변경하면 UI 로직도 함께 변경해야 함
- `onChange` → `onSubmit` 변경 시 `errors` 체크 로직 제거 필수

### 2. 버튼 비활성화 조건의 최소화
```tsx
// ❌ 나쁜 예: 너무 많은 조건
disabled={
  !field.state.value ||
  field.state.meta.errors.length > 0 ||
  field.state.meta.isTouched === false ||
  // ... 더 많은 조건
}

// ✅ 좋은 예: 최소한의 조건
disabled={
  emailCheckStatus === 'checking' ||
  !field.state.value
}
```

### 3. 중복 확인 버튼의 역할
- 형식 검증까지 하려고 하지 말 것
- 값이 있으면 클릭 가능하게 하고, API가 에러 처리
- 사용자가 시도할 수 있게 허용
