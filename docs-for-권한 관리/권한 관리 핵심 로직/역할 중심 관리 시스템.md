# 권한 중심 관리 시스템 - Authority-Centric RBAC

## 핵심 원칙

### 1. 권한(Authority)이 중심
- **`user_authority` 테이블이 단일 진실 공급원(Single Source of Truth)**
- 역할(Role)은 권한의 묶음을 표현하는 보조 개념 (템플릿)
- JWT 토큰의 `authorities` 배열에 권한 목록 포함

### 2. 권한 확인의 핵심 질문
> **"이 사용자가 X 권한을 가지고 있는가?"**

이 간단한 질문으로 대부분의 권한 제어를 구현할 수 있습니다.

## 권한 검증 구현

### 백엔드 (Spring Security)

#### 1) 페이지 수준 - ROLE 기반
**SecurityConfig에서 URL 패턴 제어**
```java
http.authorizeHttpRequests(auth -> auth
    // 관리자 페이지는 ROLE_ADMIN만 접근
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    // 나머지 인증된 사용자만 접근
    .requestMatchers("/api/**").authenticated()
)
```

#### 2) 메서드 수준 - Authority 기반
**@PreAuthorize로 세밀한 제어**
```java
@PreAuthorize("hasAuthority('USER:DELETE')")
public void deleteUser(Long id) {
    // 사용자 삭제 로직
}

@PreAuthorize("hasAuthority('PROJECT:CREATE')")
public Project createProject(ProjectRequest request) {
    // 프로젝트 생성 로직
}
```

### 프론트엔드 (React Router + TanStack Router)

#### 1) 페이지 수준 - ROLE 기반
**관리자 페이지 접근 제어**
```typescript
// routes/index.tsx
const requireRole = (requiredRole: string) => {
  const auth = authStore.state

  if (auth.user?.role !== requiredRole) {
    toast.error('접근 권한이 없습니다')
    throw redirect({ to: '/dashboard' })
  }
}

const adminUsersRoute = createRoute({
  path: '/admin/users',
  beforeLoad: () => requireRole('ROLE_ADMIN'),
  component: UsersPage,
})
```

#### 2) UI 수준 - Authority 기반
**버튼/기능 표시 여부 세밀하게 제어**
```tsx
// 사용자 관리 페이지
function UsersPage() {
  const { user } = useAuthStore()

  return (
    <div>
      {/* 권한 있는 사용자만 삭제 버튼 표시 */}
      {user.authorities.includes('USER:DELETE') && (
        <Button onClick={handleDelete}>삭제</Button>
      )}

      {/* 권한 있는 사용자만 수정 버튼 표시 */}
      {user.authorities.includes('USER:UPDATE') && (
        <Button onClick={handleEdit}>수정</Button>
      )}
    </div>
  )
}
```

### 권한 조회 로직
```java
// JwtTokenProvider.java
public String generateAccessToken(Long userId, String email, String role) {
    // user_authority 테이블에서 직접 조회
    List<String> authorities = userAuthorityRepository
        .findAuthorityNamesByUserId(userId);

    return Jwts.builder()
        .subject(email)
        .claim("userId", userId)
        .claim("role", role)
        .claim("authorities", authorities)  // 권한 배열
        .signWith(secretKey)
        .compact();
}
```

## 역할(Role)의 역할

### 역할은 "권한 템플릿"
- 여러 권한의 묶음을 표현하는 보조 개념
- 실제로는 역할의 권한들이 개별적으로 `user_authority`에 저장됨

### 역할 부여 로직
```java
// RoleService.java
@Transactional
public void grantRoleToUser(Long userId, Long roleId, Long grantedBy) {
    // 1. 역할의 권한 ID 목록 조회
    List<Long> authorityIds = roleRepository.findAuthorityIdsByRoleId(roleId);

    // 2. 각 권한을 user_authority 테이블에 추가
    for (Long authorityId : authorityIds) {
        if (!userAuthorityRepository.exists(userId, authorityId)) {
            UserAuthority userAuthority = UserAuthority.builder()
                .userId(userId)
                .authorityId(authorityId)
                .grantedAt(LocalDateTime.now())
                .grantedBy(grantedBy)
                .notes("Granted from role: " + roleId)
                .build();
            userAuthorityRepository.grant(userAuthority);
        }
    }
}
```

### 역할 회수 로직
```java
@Transactional
public void revokeRoleFromUser(Long userId, Long roleId) {
    // 이전 역할의 권한들을 user_authority에서 제거
    List<Long> authorityIds = roleRepository.findAuthorityIdsByRoleId(roleId);

    for (Long authorityId : authorityIds) {
        userAuthorityRepository.revoke(userId, authorityId);
    }
}
```

## 권한 확인 패턴 정리

### 1. 큰 범위 제어 (페이지/라우트) → ROLE
- 백엔드: `hasRole("ADMIN")`
- 프론트: `requireRole('ROLE_ADMIN')`
- **사용 시점**: 관리자 전용 페이지, 관리자 메뉴들

### 2. 세밀한 제어 (기능/버튼) → Authority
- 백엔드: `hasAuthority("USER:DELETE")`
- 프론트: `user.authorities.includes('USER:DELETE')`
- **사용 시점**: 삭제 버튼, 수정 버튼, 특정 API 호출

### 3. 비즈니스 로직 내부 판단 → ROLE (편의상 사용)
```java
if (user.isAdmin()) {
    // 관리자만 가능한 로직
} else {
    // 일반 사용자용 로직
}
```

## 데이터베이스 구조

### 핵심 테이블
```sql
-- 1. authority: 권한 정의
CREATE TABLE authority (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) UNIQUE NOT NULL,  -- 예: USER:DELETE
    description VARCHAR(255),
    category_id BIGINT,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 2. user_authority: 사용자-권한 관계 (Source of Truth)
CREATE TABLE user_authority (
    user_id BIGINT NOT NULL,
    authority_id BIGINT NOT NULL,
    granted_at TIMESTAMP DEFAULT NOW(),
    granted_by BIGINT,
    expires_at TIMESTAMP,
    notes TEXT,
    PRIMARY KEY (user_id, authority_id)
);

-- 3. role: 역할 (권한 템플릿)
CREATE TABLE role (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) UNIQUE NOT NULL,
    description VARCHAR(255),
    created_at TIMESTAMP DEFAULT NOW()
);

-- 4. role_authority: 역할-권한 관계
CREATE TABLE role_authority (
    role_id BIGINT NOT NULL,
    authority_id BIGINT NOT NULL,
    PRIMARY KEY (role_id, authority_id)
);
```

## 확장 가능성

### 현재 구조로 가능한 확장:
1. ✅ 개별 사용자 권한 부여/회수
2. ✅ 역할 기반 일괄 권한 부여
3. ✅ 권한 만료 시간 설정 (`expires_at`)
4. ✅ 권한 부여 이력 추적 (`granted_by`, `granted_at`, `notes`)
5. ✅ 페이지/기능/API 단위 권한 제어
6. ✅ 프론트/백엔드 양방향 권한 확인

### 향후 확장 가능:
- 조직(Organization) 단위 권한 관리
- 프로젝트별 권한 만료 시간 설정
- 임시 권한 승인/반려
- 권한 변경/감사 로그 시스템

## 요약

**"사용자가 X 권한을 가지고 있는가?"** 이 핵심 질문으로:
- 백엔드: `hasAuthority('권한명')`
- 프론트: `user.authorities.includes('권한명')`

대부분의 권한 제어를 구현할 수 있으며, ROLE은 큰 범위 접근 제어 또는 비즈니스 로직에서 편의상 사용합니다.
