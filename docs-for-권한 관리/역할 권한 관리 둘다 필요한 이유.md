# 역할(Role)과 권한(Authority) 관리가 둘 다 필요한 이유

## 문제 인식

### 기존 방식의 한계
현재 `ROLE_ADMIN`, `ROLE_USER` 방식은 "역할(Role)"만 사용하고 있어 세밀한 권한 제어가 불가능합니다.

```java
// 현재 방식 - 역할만으로 모든 것을 제어
@PreAuthorize("hasRole('ADMIN')")  // ADMIN이면 모든 것 가능
@PreAuthorize("hasRole('USER')")   // USER면 제한적
```

**문제점:**
1. 같은 USER 역할이라도 사용자별로 다른 권한을 주기 어려움
2. 새로운 기능 추가 시 역할을 늘려야 함 (ROLE_MANAGER, ROLE_VIEWER 등)
3. 역할이 많아지면 관리가 복잡해짐
4. 역할 계층(Role Hierarchy)도 제한적

## 올바른 설계: 역할 + 권한 분리

### 역할 (Role) - "사람의 분류"
사용자를 큰 범주로 분류하는 용도

```
ADMIN     - 관리자
USER      - 일반 사용자
MANAGER   - 프로젝트 매니저
VIEWER    - 조회 전용
```

### 권한 (Authority) - "실제 행위"
시스템에서 수행할 수 있는 구체적인 동작

```
메뉴 접근
├─ MENU:ADMIN:READ
├─ MENU:ADMIN:WRITE
├─ MENU:DASHBOARD:READ
└─ MENU:PROJECT:READ

프로젝트 관리
├─ PROJECT:CREATE
├─ PROJECT:READ
├─ PROJECT:UPDATE
├─ PROJECT:DELETE
└─ PROJECT:ASSIGN

사용자 관리
├─ USER:CREATE
├─ USER:READ
├─ USER:UPDATE
├─ USER:DELETE
└─ USER:ROLE_CHANGE

기타
├─ DASHBOARD:VIEW
├─ REPORT:EXPORT
└─ SETTINGS:MANAGE
```

### 역할-권한 매핑
역할에 따라 권한을 부여하되, 개별 사용자별로도 추가 권한 부여 가능

```
ADMIN → [모든 권한]

MANAGER → [
  MENU:DASHBOARD:READ,
  MENU:PROJECT:READ,
  PROJECT:CREATE,
  PROJECT:READ,
  PROJECT:UPDATE,
  PROJECT:ASSIGN,
  REPORT:EXPORT
]

USER → [
  MENU:DASHBOARD:READ,
  MENU:PROJECT:READ,
  PROJECT:READ,
  DASHBOARD:VIEW
]

VIEWER → [
  MENU:DASHBOARD:READ,
  DASHBOARD:VIEW
]
```

## 구현 방식

### 1. 데이터베이스 설계

```sql
-- 역할 테이블 (기존 users.role 활용)
-- ADMIN, USER, MANAGER, VIEWER

-- 권한 테이블
CREATE TABLE authority (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,  -- 예: PROJECT:CREATE
    description VARCHAR(255),
    category VARCHAR(50),                -- 예: MENU, PROJECT, USER
    created_at TIMESTAMP DEFAULT NOW()
);

-- 역할-권한 매핑 테이블
CREATE TABLE role_authority (
    role VARCHAR(50) NOT NULL,           -- ADMIN, USER 등
    authority_id BIGINT NOT NULL,
    PRIMARY KEY (role, authority_id),
    FOREIGN KEY (authority_id) REFERENCES authority(id)
);

-- 사용자별 추가 권한 테이블 (선택적)
CREATE TABLE user_authority (
    user_id BIGINT NOT NULL,
    authority_id BIGINT NOT NULL,
    granted BOOLEAN DEFAULT TRUE,        -- true: 추가 부여, false: 제거
    PRIMARY KEY (user_id, authority_id),
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (authority_id) REFERENCES authority(id)
);

-- 권한 계층 테이블 (기존 role_hierarchy를 authority_hierarchy로 변경)
CREATE TABLE authority_hierarchy (
    parent_authority_id BIGINT NOT NULL,
    child_authority_id BIGINT NOT NULL,
    PRIMARY KEY (parent_authority_id, child_authority_id),
    FOREIGN KEY (parent_authority_id) REFERENCES authority(id),
    FOREIGN KEY (child_authority_id) REFERENCES authority(id)
);
```

### 2. JWT 토큰 구조

```json
{
  "sub": "user@example.com",
  "role": "USER",
  "roles": ["USER"],                    // 역할 배열 (계층 포함)
  "authorities": [                      // 권한 배열
    "MENU:DASHBOARD:READ",
    "MENU:PROJECT:READ",
    "PROJECT:READ",
    "DASHBOARD:VIEW"
  ],
  "type": "ACCESS",
  "iat": 1234567890,
  "exp": 1234571490
}
```

### 3. Spring Security 사용 예시

```java
// 역할 기반 (큰 범주)
@PreAuthorize("hasRole('ADMIN')")
public void adminOnlyMethod() { }

// 권한 기반 (세밀한 제어)
@PreAuthorize("hasAuthority('PROJECT:CREATE')")
public void createProject() { }

@PreAuthorize("hasAuthority('USER:DELETE')")
public void deleteUser() { }

// 복합 조건
@PreAuthorize("hasRole('MANAGER') and hasAuthority('PROJECT:DELETE')")
public void deleteProject() { }

// OR 조건
@PreAuthorize("hasAuthority('PROJECT:UPDATE') or hasAuthority('PROJECT:CREATE')")
public void saveProject() { }
```

### 4. 프론트엔드 권한 체크

```typescript
// 역할 체크 (UI 표시용)
if (user.role === 'ADMIN') {
  // 관리자 UI 표시
}

// 권한 체크 (기능 제어)
const canCreateProject = user.authorities.includes('PROJECT:CREATE')
const canDeleteUser = user.authorities.includes('USER:DELETE')
const canAccessAdminMenu = user.authorities.includes('MENU:ADMIN:READ')

// React Router 예시
{
  path: '/admin/users',
  beforeLoad: () => requireAuthority('USER:READ')
}

{
  path: '/projects/new',
  beforeLoad: () => requireAuthority('PROJECT:CREATE')
}
```

## 장점

### 1. 유연성
- 같은 역할이라도 사용자별로 다른 권한 부여 가능
- 임시로 특정 사용자에게만 권한 추가/제거 가능

### 2. 확장성
- 새로운 기능 추가 시 권한만 추가하면 됨
- 역할을 늘리지 않아도 됨

### 3. 명확성
- 역할: "이 사람은 누구인가" (관리자, 사용자, 매니저)
- 권한: "이 사람은 무엇을 할 수 있는가" (프로젝트 생성, 사용자 삭제)

### 4. 세밀한 제어
- 페이지별, 기능별, 버튼별로 권한 제어 가능
- 메뉴 표시/숨김도 권한 기반으로 제어

### 5. 감사(Audit) 용이
- 누가 어떤 권한을 가지고 있는지 명확히 추적
- 권한 변경 이력 관리 가능

## 구현 우선순위

1. **Phase 1: 기본 구조**
   - `authority`, `role_authority` 테이블 생성
   - 기본 권한 데이터 삽입
   - JWT에 `authorities` 배열 추가

2. **Phase 2: 백엔드 통합**
   - Authority 조회 서비스 구현
   - JWT 생성 시 authorities 포함
   - Spring Security `@PreAuthorize` 적용

3. **Phase 3: 프론트엔드 통합**
   - JWT에서 authorities 추출
   - 권한 기반 라우팅 가드
   - 권한 기반 UI 표시/숨김

4. **Phase 4: 고급 기능** (선택)
   - 사용자별 추가 권한 (`user_authority`)
   - 권한 계층 (`authority_hierarchy`)
   - 관리자 페이지에서 권한 관리 UI

## 권한 네이밍 컨벤션

```
카테고리:리소스:액션

예시:
MENU:ADMIN:READ         - 관리자 메뉴 읽기
MENU:ADMIN:WRITE        - 관리자 메뉴 쓰기
PROJECT:CREATE          - 프로젝트 생성
PROJECT:READ            - 프로젝트 읽기
PROJECT:UPDATE          - 프로젝트 수정
PROJECT:DELETE          - 프로젝트 삭제
USER:MANAGE             - 사용자 관리
DASHBOARD:VIEW          - 대시보드 조회
REPORT:EXPORT           - 리포트 내보내기
SETTINGS:SYSTEM:UPDATE  - 시스템 설정 수정
```

## 결론

역할(Role)과 권한(Authority)을 분리하면:
- **역할**: 사용자 분류 및 UI 표시
- **권한**: 실제 기능 접근 제어

이렇게 함으로써 유연하고 확장 가능하며 세밀한 권한 관리 시스템을 구축할 수 있습니다.
