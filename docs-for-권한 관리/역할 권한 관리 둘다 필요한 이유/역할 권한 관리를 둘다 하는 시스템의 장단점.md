# 역할(Role)과 권한(Authority) 분리 시스템의 장단점

## 개요

본 프로젝트에서는 **Role(역할)**과 **Authority(권한)**을 분리하여 관리하는 RBAC(Role-Based Access Control) 시스템을 구현했습니다.

---

## 시스템 구조

### 기본 개념

- **Role (역할)**: 사용자의 신분/직책 (예: ROLE_ADMIN, ROLE_USER)
- **Authority (권한)**: 실제 수행 가능한 작업 (예: USER:READ, MENU:ADMIN:WRITE)
- **역할-권한 매핑**: N:M 관계로 하나의 역할이 여러 권한을 가질 수 있음

### 데이터베이스 구조

```sql
-- 권한 테이블
CREATE TABLE authority (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,
    description VARCHAR(255),
    category VARCHAR(50)
);

-- 역할-권한 매핑 테이블
CREATE TABLE role_authority (
    role VARCHAR(50) NOT NULL,
    authority_id BIGINT NOT NULL,
    PRIMARY KEY (role, authority_id),
    FOREIGN KEY (authority_id) REFERENCES authority(id)
);
```

### JWT 토큰 구조

```json
{
  "sub": "user@example.com",
  "role": "ROLE_ADMIN",
  "roles": ["ROLE_ADMIN", "ROLE_USER"],
  "authorities": [
    "MENU:ADMIN:READ",
    "MENU:ADMIN:WRITE",
    "USER:READ",
    "USER:WRITE",
    "PROJECT:CREATE",
    "..."
  ]
}
```

---

## 장점 (Advantages)

### 1. 세밀한 권한 제어 (Fine-Grained Access Control)

**역할만 사용하는 경우:**
```typescript
// ❌ 역할 기반만 사용
if (user.role === 'ROLE_ADMIN') {
  // 관리자는 모든 기능에 접근 가능
  // 너무 광범위한 권한
}
```

**역할 + 권한 분리:**
```typescript
// ✅ 권한 기반 세밀한 제어
if (hasAuthority('USER:READ')) {
  // 사용자 조회만 가능
}
if (hasAuthority('USER:WRITE')) {
  // 사용자 수정 가능
}
```

### 2. 유연한 권한 관리

- 역할은 유지하면서 권한만 조정 가능
- 예: ROLE_MANAGER에게 일부 관리자 권한 부여
- 신규 기능 추가 시 기존 역할에 새 권한만 매핑

```sql
-- 예: 매니저에게 프로젝트 생성 권한 추가
INSERT INTO role_authority (role, authority_id)
SELECT 'ROLE_MANAGER', id FROM authority WHERE name = 'PROJECT:CREATE';
```

### 3. 확장성 (Scalability)

- 새로운 권한 추가가 용이
- 역할 계층 구조와 독립적
- 마이크로서비스 환경에서 권한 공유 가능

```sql
-- 신규 기능 권한 추가
INSERT INTO authority (name, description, category)
VALUES ('REPORT:EXPORT', '리포트 내보내기', 'REPORT');

-- 특정 역할에만 매핑
INSERT INTO role_authority (role, authority_id)
SELECT 'ROLE_ADMIN', id FROM authority WHERE name = 'REPORT:EXPORT';
```

### 4. 보안 강화

- 최소 권한 원칙(Principle of Least Privilege) 적용 용이
- 권한별 세부 로깅 가능
- API 엔드포인트별 권한 체크

```java
// Spring Security 예시
@PreAuthorize("hasAuthority('USER:DELETE')")
public void deleteUser(Long id) {
    // 특정 권한이 있는 사용자만 실행 가능
}
```

### 5. 감사(Audit) 및 추적

- 어떤 권한으로 어떤 작업을 했는지 명확히 추적
- 역할만으로는 파악하기 어려운 세부 작업 로그 가능

```java
log.info("User {} executed action with authority: {}",
    user.getEmail(), "USER:DELETE");
```

### 6. 역할의 의미적 명확성

- 역할: "누구인가?" (신분/직책)
- 권한: "무엇을 할 수 있는가?" (기능)
- UI에서 역할 표시, 기능 제어는 권한으로 분리

```typescript
// UI: 역할 표시
<Badge>{user.role}</Badge>  // "관리자", "매니저", "일반 사용자"

// 기능: 권한 체크
{hasAuthority('USER:WRITE') && <Button>사용자 수정</Button>}
```

## 단점 (Disadvantages)

### 1. 구현 복잡도 증가

**역할만 사용:**
```typescript
// 간단
if (user.role === 'ROLE_ADMIN') {
  // 접근 허용
}
```

**역할 + 권한:**
```typescript
// 복잡
// 1. DB에서 역할-권한 조회
// 2. JWT에 권한 배열 포함
// 3. 프론트엔드에서 권한 파싱
// 4. 권한 체크 함수 구현
if (hasAuthority('USER:READ')) {
  // 접근 허용
}
```

### 2. 데이터베이스 복잡도

- 추가 테이블 필요 (authority, role_authority)
- 조인 쿼리 증가
- 권한 데이터 초기화 필요

```sql
-- 매번 조인 필요
SELECT a.name
FROM authority a
INNER JOIN role_authority ra ON a.id = ra.authority_id
WHERE ra.role = 'ROLE_ADMIN';
```

### 3. 성능 오버헤드

- JWT 토큰 크기 증가 (authorities 배열)
- 권한 조회 쿼리 추가
- 메모리 사용량 증가

```json
// JWT 크기 비교
// Role만: ~200 bytes
{
  "sub": "user@example.com",
  "role": "ROLE_ADMIN"
}

// Role + Authorities: ~800+ bytes
{
  "sub": "user@example.com",
  "role": "ROLE_ADMIN",
  "roles": ["ROLE_ADMIN", "ROLE_USER"],
  "authorities": ["AUTH1", "AUTH2", "...16개"]  // 더 많은 데이터
}
```

### 4. 학습 곡선

- 팀원들이 Role vs Authority 개념 이해 필요
- 권한 명명 규칙 학습 (CATEGORY:RESOURCE:ACTION)
- 프론트-백엔드 권한 동기화 이해 필요

### 5. 초기 설정 비용

- 권한 정의 및 분류 작업
- 역할-권한 매핑 설계
- 기존 시스템 마이그레이션 비용

```sql
-- 초기 권한 데이터 삽입
INSERT INTO authority (name, description, category) VALUES
('MENU:ADMIN:READ', '관리자 메뉴 조회', 'MENU'),
('MENU:ADMIN:WRITE', '관리자 메뉴 수정', 'MENU'),
-- ... 16개 권한
```

### 6. 유지보수 부담

- 권한 변경 시 DB + 코드 모두 수정
- 권한 충돌/중복 관리 필요
- 사용하지 않는 권한 정리 필요

## 권장 사용 시나리오

### 역할 + 권한 분리를 사용해야 할 때 ✅

1. **대규모 엔터프라이즈 시스템**
   - 다양한 역할과 복잡한 권한 구조
   - 예: ERP, CRM, 프로젝트 관리 시스템

2. **보안이 중요한 시스템**
   - 금융, 의료, 정부 시스템
   - 감사 추적 필요

3. **다양한 사용자 그룹**
   - 여러 조직/부서가 사용
   - 역할 간 권한이 유동적

4. **장기 운영 시스템**
   - 지속적인 기능 추가
   - 유연한 권한 관리 필요

### 역할만 사용해도 되는 경우 ❌

1. **소규모 프로젝트**
   - 사용자 수 < 100명
   - 역할 2-3개 (관리자, 사용자)

2. **단순한 권한 구조**
   - 역할과 권한이 1:1 대응
   - 권한 변경 빈도 낮음

3. **빠른 프로토타이핑**
   - MVP 단계
   - 빠른 개발 우선

## 구현 전략

### 1. 점진적 도입

```typescript
// Phase 1: Role만 사용 (초기)
if (user.role === 'ROLE_ADMIN') { /* ... */ }

// Phase 2: Authority 추가 (확장)
if (hasAuthority('USER:WRITE')) { /* ... */ }

// Phase 3: 완전 전환 (최종)
// Role은 UI 표시용, Authority는 기능 제어용
```

### 2. 하이브리드 접근

```typescript
// 페이지 접근: Authority 기반
const adminUsersRoute = createRoute({
  path: '/admin/users',
  beforeLoad: () => requireAuthority('USER:READ'),
})

// UI 표시: Role 기반
<Badge>{user.role}</Badge>  // "관리자"

// 기능 제어: Authority 기반
{hasAuthority('USER:WRITE') && <EditButton />}
```

### 3. 권한 네이밍 컨벤션

```
패턴: CATEGORY:RESOURCE:ACTION

카테고리 (선택):
- MENU: 메뉴 관련
- USER: 사용자 관련
- PROJECT: 프로젝트 관련

리소스:
- ADMIN: 관리 기능
- DASHBOARD: 대시보드
- REPORT: 리포트

액션:
- READ: 조회
- WRITE: 생성/수정
- DELETE: 삭제
- EXPORT: 내보내기

예시:
- MENU:ADMIN:READ
- USER:WRITE
- PROJECT:CREATE
- REPORT:EXPORT
```

## 본 프로젝트의 선택

### 선택 이유

1. **엔터프라이즈 급 시스템**을 목표로 함
2. **확장 가능성**을 고려한 설계
3. **보안 및 감사** 기능 필요
4. **다양한 역할과 권한** 조합 필요

### 트레이드오프 수용

- 초기 구현 복잡도 증가 ← **장기적 유연성 확보**
- 성능 오버헤드 발생 ← **세밀한 권한 제어**
- 학습 곡선 ← **체계적인 권한 관리**

### 결론

> "초기 복잡도는 증가하지만, 시스템의 확장성과 보안성을 위해 역할과 권한을 분리하는 것이 장기적으로 더 유리하다."
