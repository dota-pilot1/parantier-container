# 권한 관리의 포함 관계 자체를 관리

## 개요

권한 계층 구조를 데이터베이스 테이블로 관리하여, 코드 수정 없이 권한 관계를 동적으로 확장할 수 있는 시스템

## 문제 상황

### 초기 문제: 하드코딩된 권한 필터링

```sql
-- ❌ 나쁜 방식: 권한이 추가될 때마다 SQL 수정 필요
WHERE required_role IS NULL
   OR required_role = #{role}
   OR (#{role} = 'ADMIN' AND required_role = 'USER')
   OR (#{role} = 'ADMIN' AND required_role = 'MANAGER')  -- 새 권한 추가 시
   OR (#{role} = 'SUPER_ADMIN' AND required_role = 'ADMIN')  -- 또 추가...
```

**문제점:**
- 권한 추가할 때마다 코드 수정 + 배포 필요
- SQL이 점점 복잡해짐
- 유지보수 어려움

## 해결책: 권한 계층 테이블

### 1. 테이블 설계

```sql
-- 권한 계층 테이블
CREATE TABLE role_hierarchy (
    parent_role VARCHAR(50) NOT NULL,  -- 상위 권한 (더 강한 권한)
    child_role VARCHAR(50) NOT NULL,   -- 하위 권한 (포함되는 권한)
    PRIMARY KEY (parent_role, child_role)
);

COMMENT ON TABLE role_hierarchy IS '권한 계층 구조 (parent_role이 child_role의 권한을 포함)';
```

### 2. 데이터 예시

#### 기본 구조 (2단계)
```sql
-- ADMIN은 USER 권한을 포함
INSERT INTO role_hierarchy (parent_role, child_role)
VALUES ('ADMIN', 'USER');
```

| parent_role | child_role |
|-------------|-----------|
| ADMIN       | USER      |

**의미**: ADMIN 권한을 가진 사용자는 USER 권한이 필요한 메뉴도 볼 수 있다

#### 확장된 구조 (4단계)
```sql
INSERT INTO role_hierarchy (parent_role, child_role) VALUES
    ('SUPER_ADMIN', 'ADMIN'),
    ('ADMIN', 'MANAGER'),
    ('MANAGER', 'USER');
```

| parent_role  | child_role |
|--------------|-----------|
| SUPER_ADMIN  | ADMIN     |
| ADMIN        | MANAGER   |
| MANAGER      | USER      |

## 작동 원리: 재귀 CTE

### SQL 쿼리 구조

```sql
WITH RECURSIVE
    menu_tree AS (
        -- 메뉴 트리 구조 조회 (생략)
        ...
    ),
    -- 현재 권한이 접근 가능한 모든 하위 권한 (재귀 CTE)
    accessible_roles AS (
        -- 1단계: 본인 권한
        SELECT #{role} as role

        UNION ALL

        -- 2단계: 하위 권한을 재귀적으로 찾기
        SELECT rh.child_role
        FROM role_hierarchy rh
        INNER JOIN accessible_roles ar ON rh.parent_role = ar.role
    )
SELECT * FROM menu_tree
WHERE required_role IS NULL
   OR required_role IN (SELECT role FROM accessible_roles)
ORDER BY parent_id NULLS FIRST, order_num
```

### 실행 과정 예시

#### Case 1: ADMIN 로그인

```
1회전: ADMIN (본인)
2회전: role_hierarchy에서 parent_role='ADMIN' 찾기 → USER 발견
3회전: role_hierarchy에서 parent_role='USER' 찾기 → 없음 → 종료

결과: accessible_roles = ['ADMIN', 'USER']
```

**메뉴 필터링:**
```sql
WHERE required_role IN ('ADMIN', 'USER')
```
→ ADMIN 메뉴 + USER 메뉴 모두 조회 ✅

#### Case 2: SUPER_ADMIN 로그인 (4단계 계층)

```
1회전: SUPER_ADMIN (본인)
2회전: parent_role='SUPER_ADMIN' → ADMIN 발견
3회전: parent_role='ADMIN' → MANAGER 발견
4회전: parent_role='MANAGER' → USER 발견
5회전: parent_role='USER' → 없음 → 종료

결과: accessible_roles = ['SUPER_ADMIN', 'ADMIN', 'MANAGER', 'USER']
```

**메뉴 필터링:**
```sql
WHERE required_role IN ('SUPER_ADMIN', 'ADMIN', 'MANAGER', 'USER')
```
→ 모든 메뉴 조회 ✅

#### Case 3: USER 로그인

```
1회전: USER (본인)
2회전: parent_role='USER' → 없음 → 종료

결과: accessible_roles = ['USER']
```

**메뉴 필터링:**
```sql
WHERE required_role IN ('USER')
```
→ USER 메뉴만 조회 (ADMIN, MANAGER 메뉴 차단) ✅

## 장점

### 1. 확장성
```sql
-- 새 권한 추가 시 테이블에 행만 추가
INSERT INTO role_hierarchy (parent_role, child_role)
VALUES ('ADMIN', 'MODERATOR');
```
→ **코드 수정 불필요, 배포 불필요**

### 2. 유지보수성
- SQL 쿼리는 한 번 작성 후 변경 불필요
- 권한 관계는 데이터로만 관리
- 복잡한 계층 구조도 쉽게 표현 가능

### 3. 유연성
다양한 계층 구조 지원:
- 선형: A → B → C → D
- 다중 상속: A → B, A → C (한 권한이 여러 하위 권한 포함 가능)

### 4. 성능
- 재귀 CTE는 권한 계층 깊이만큼만 실행 (보통 3-5단계)
- 한 번 실행되고 결과가 메모리에 캐시됨
- 전체 메뉴 스캔보다 훨씬 효율적
- 예상 성능: ~10-15ms (충분히 빠름)

## 단점

### 1. 복잡도 증가
- 재귀 CTE에 대한 이해 필요
- SQL 초보자에게는 어려울 수 있음

### 2. 디버깅
- 권한 관계 오류 시 테이블 데이터 확인 필요
- 순환 참조 주의 (A → B → A)

## 파일 위치

### 테이블 생성
```
직접 PostgreSQL에서 실행
→ Flyway는 현재 프로젝트에서 사용하지 않음 (버전 충돌 문제로 인해 비활성화)
→ 더 안정적인 직접 DB 조작 방식 채택
```

**테이블 생성 명령어:**
```sql
docker exec palantier-postgres psql -U palantier_user -d palantier -c "
CREATE TABLE IF NOT EXISTS role_hierarchy (
    parent_role VARCHAR(50) NOT NULL,
    child_role VARCHAR(50) NOT NULL,
    PRIMARY KEY (parent_role, child_role)
);

INSERT INTO role_hierarchy (parent_role, child_role) VALUES ('ADMIN', 'USER')
ON CONFLICT DO NOTHING;

COMMENT ON TABLE role_hierarchy IS '권한 계층 구조 (parent_role이 child_role의 권한을 포함)';
"
```

### SQL 쿼리
```
parantier-api/src/main/resources/mybatis/mapper/MenuMapper.xml
→ findMenuTreeByRole 메서드
```

## 사용 사례

### 새 권한 추가하기

1. **데이터베이스에 권한 관계 추가**
```bash
docker exec palantier-postgres psql -U palantier_user -d palantier -c "
INSERT INTO role_hierarchy (parent_role, child_role)
VALUES ('ADMIN', 'EDITOR');
"
```

2. **끝!** (코드 수정 불필요, 서버 재시작 불필요)

### 권한 계층 확인하기

```sql
-- ADMIN이 접근 가능한 모든 권한 확인
WITH RECURSIVE accessible_roles AS (
    SELECT 'ADMIN' as role
    UNION ALL
    SELECT rh.child_role
    FROM role_hierarchy rh
    INNER JOIN accessible_roles ar ON rh.parent_role = ar.role
)
SELECT * FROM accessible_roles;
```

결과:
```
role
------
ADMIN
USER
```

## 베스트 프랙티스

### 1. 순환 참조 방지
```sql
-- ❌ 나쁜 예: 순환 참조
INSERT INTO role_hierarchy VALUES ('ADMIN', 'USER');
INSERT INTO role_hierarchy VALUES ('USER', 'ADMIN');  -- 무한 루프!
```

### 2. 권한 네이밍 컨벤션
- 명확한 이름 사용: `SUPER_ADMIN`, `ADMIN`, `USER`
- 접두사 통일: `ROLE_` 없이 저장 (컨트롤러에서 제거 후 사용)

### 3. 문서화
- 권한 계층 구조를 다이어그램으로 문서화
- 예시:
```
SUPER_ADMIN
    └── ADMIN
        ├── MANAGER
        │   └── USER
        └── EDITOR
            └── USER
```

## 대안과 비교

### 하드코딩 방식
```sql
WHERE required_role = #{role}
   OR (#{role} = 'ADMIN' AND required_role = 'USER')
```
- 장점: 이해하기 쉬움
- 단점: 확장성 없음

### Spring Security ACL
- 장점: 엔터프라이즈급 권한 관리
- 단점: 설정 복잡, 오버 엔지니어링 가능성

### 재귀 CTE (현재 방식) ✅
- 장점: 확장성 + 성능 + 유지보수성
- 단점: SQL 이해도 필요
- **추천 사용 케이스**: 중대형 프로젝트

## 결론

권한 계층을 테이블로 관리하는 것은:
- ✅ 확장 가능한 시스템 구축
- ✅ 유지보수 비용 절감
- ✅ 성능 문제 없음
- ⚠️ 초기 학습 곡선 존재

**트레이드오프**: 복잡도는 올라가지만, 장기적으로 더 나은 선택
