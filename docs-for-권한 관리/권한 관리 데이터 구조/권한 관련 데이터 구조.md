# 권한 관리 데이터 구조

## 개요

Palantier 프로젝트의 권한 관리 시스템은 **역할 기반 권한 + 사용자별 개별 권한**을 결합한 하이브리드 모델을 사용합니다.

## 데이터 흐름

```
1. Authority (권한 마스터) 생성
   ↓
2. Role-Authority 매핑 (역할에 권한 부여)
   ↓
3. User-Authority 매핑 (사용자에게 개별 권한 부여)
   ↓
4. 최종 권한 = 역할 권한 + 개별 권한
```

---

## 1. 데이터베이스 테이블 구조

### 1.1 `authority` (권한 마스터 테이블)

시스템에서 사용 가능한 모든 권한의 정의

```sql
CREATE TABLE authority (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL UNIQUE,
    description TEXT,
    category VARCHAR(100),
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

**예시 데이터:**
```sql
id | name                  | description            | category
1  | MENU:ADMIN:READ      | 관리자 메뉴 읽기       | MENU
2  | MENU:ADMIN:WRITE     | 관리자 메뉴 쓰기       | MENU
3  | PROJECT:CREATE       | 프로젝트 생성          | PROJECT
4  | PROJECT:DELETE       | 프로젝트 삭제          | PROJECT
5  | USER:MANAGE          | 사용자 관리            | USER
```

**특징:**
- 권한은 먼저 이 테이블에 정의되어야 함
- `name`은 고유하며 권한 식별자로 사용
- `category`로 권한을 그룹화

---

### 1.2 `role_authority` (역할-권한 매핑 테이블)

특정 역할(Role)에 부여된 권한들

```sql
CREATE TABLE role_authority (
    role VARCHAR(50) NOT NULL,
    authority_id BIGINT NOT NULL,
    PRIMARY KEY (role, authority_id),
    FOREIGN KEY (authority_id) REFERENCES authority(id) ON DELETE CASCADE
);
```

**예시 데이터:**
```sql
role        | authority_id
ROLE_ADMIN  | 1  -- MENU:ADMIN:READ
ROLE_ADMIN  | 2  -- MENU:ADMIN:WRITE
ROLE_ADMIN  | 3  -- PROJECT:CREATE
ROLE_ADMIN  | 4  -- PROJECT:DELETE
ROLE_ADMIN  | 5  -- USER:MANAGE
ROLE_USER   | 1  -- MENU:ADMIN:READ
ROLE_USER   | 3  -- PROJECT:CREATE
```

**특징:**
- 역할에 속한 모든 사용자가 자동으로 이 권한들을 가짐
- 역할 단위로 권한을 일괄 관리

---

### 1.3 `user_authority` (사용자별 개별 권한 테이블)

특정 사용자에게 개별적으로 부여된 권한

```sql
CREATE TABLE user_authority (
    user_id BIGINT NOT NULL,
    authority_id BIGINT NOT NULL,
    granted_at TIMESTAMP NOT NULL DEFAULT NOW(),
    granted_by BIGINT,
    expires_at TIMESTAMP,
    notes TEXT,
    PRIMARY KEY (user_id, authority_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (authority_id) REFERENCES authority(id) ON DELETE CASCADE,
    FOREIGN KEY (granted_by) REFERENCES users(id) ON DELETE SET NULL
);

CREATE INDEX idx_user_authority_user_id ON user_authority(user_id);
CREATE INDEX idx_user_authority_expires_at ON user_authority(expires_at);
```

**예시 데이터:**
```sql
user_id | authority_id | granted_at          | granted_by | expires_at          | notes
5       | 3            | 2024-01-15 10:00:00 | 1         | NULL                | 임시 프로젝트 생성 권한
5       | 4            | 2024-01-15 10:00:00 | 1         | 2024-12-31 23:59:59 | 2024년까지만 유효
8       | 5            | 2024-02-01 14:30:00 | 1         | 2024-06-30 23:59:59 | 6개월 임시 사용자 관리 권한
```

**필드 설명:**
- `user_id`: 권한을 받는 사용자 ID
- `authority_id`: 부여된 권한 ID (authority 테이블 참조)
- `granted_at`: 권한 부여 시각
- `granted_by`: 권한을 부여한 관리자 ID
- `expires_at`: 권한 만료 시각 (NULL이면 영구)
- `notes`: 권한 부여 사유나 메모

**특징:**
- 역할과 무관하게 특정 사용자에게만 권한 부여 가능
- 만료 기능으로 임시 권한 부여 가능
- 권한 부여 이력 추적 가능

## 2. 권한 계산 로직

### 최종 사용자 권한 = 역할 권한 + 개별 권한

```java
public List<String> getUserEffectiveAuthorities(Long userId, String role) {
    // 1. 역할 기반 권한 조회
    List<String> roleAuthorities = authorityRepository.findAuthorityNamesByRole(role);

    // 2. 개별 사용자 권한 조회 (유효한 것만)
    List<String> userAuthorities = userAuthorityRepository.findValidByUserId(userId)
            .stream()
            .map(ua -> ua.getAuthority().getName())
            .collect(Collectors.toList());

    // 3. 중복 제거 후 병합
    return Stream.concat(roleAuthorities.stream(), userAuthorities.stream())
            .distinct()
            .collect(Collectors.toList());
}
```

**예시:**
```
사용자 ID: 5
역할: ROLE_USER

ROLE_USER 권한:
- MENU:ADMIN:READ
- PROJECT:CREATE

개별 권한 (user_authority):
- PROJECT:DELETE (만료 안됨)
- USER:MANAGE (만료됨 - 제외)

최종 권한:
- MENU:ADMIN:READ (역할에서)
- PROJECT:CREATE (역할에서)
- PROJECT:DELETE (개별 권한에서)
```

## 3. 데이터 관계도

```
users (사용자)
  ├─ role (역할) ─→ role_authority (역할-권한 매핑)
  │                        ↓
  │                   authority (권한 마스터)
  │                        ↑
  └─ user_authority (사용자별 개별 권한)
```

## 4. 사용 시나리오

### 시나리오 1: 새로운 권한 추가

1. **권한 마스터 생성**
   ```http
   POST /api/authorities
   {
     "name": "REPORT:EXPORT",
     "description": "리포트 내보내기",
     "category": "REPORT"
   }
   ```

2. **역할에 권한 부여** (선택)
   ```http
   PUT /api/authorities/role/ROLE_ADMIN/mapping
   {
     "authorityIds": [1, 2, 3, 4, 5, 6]  // 6번이 새로 추가된 REPORT:EXPORT
   }
   ```

3. **특정 사용자에게 개별 권한 부여** (선택)
   ```http
   POST /api/authorities/user/5
   {
     "authorityId": 6,
     "expiresAt": "2024-12-31T23:59:59",
     "notes": "2024년 말까지 리포트 권한 테스트"
   }
   ```

### 시나리오 2: 임시 권한 부여

프로젝트 매니저가 아닌 일반 사용자에게 한 달간 프로젝트 삭제 권한을 부여:

```http
POST /api/authorities/user/8
{
  "authorityId": 4,  // PROJECT:DELETE
  "expiresAt": "2024-04-30T23:59:59",
  "notes": "프로젝트 정리 작업을 위한 임시 권한"
}
```

### 시나리오 3: 권한 회수

```http
DELETE /api/authorities/user/8/authority/4
```

## 5. 백엔드 구현 파일

### Domain Layer
- `UserAuthority.java` - 사용자 권한 엔티티
- `UserAuthorityRepository.java` - 리포지토리 인터페이스

### Infrastructure Layer
- `UserAuthorityMapper.java` - MyBatis 매퍼 인터페이스
- `UserAuthorityRepositoryImpl.java` - 리포지토리 구현
- `UserAuthorityMapper.xml` - SQL 쿼리

### Service Layer
- `AuthorityService.java` - 권한 비즈니스 로직
  - `getUserAuthorities()` - 사용자의 모든 개별 권한 조회
  - `getValidUserAuthorities()` - 유효한 권한만 조회
  - `getUserEffectiveAuthorities()` - 역할 + 개별 권한 합산
  - `grantUserAuthority()` - 권한 부여
  - `revokeUserAuthority()` - 권한 회수
  - `revokeAllUserAuthorities()` - 모든 개별 권한 회수

### Controller Layer
- `AuthorityController.java` - REST API 엔드포인트

### DTOs
- `GrantUserAuthorityRequest.java` - 권한 부여 요청
- `UserAuthorityResponse.java` - 권한 응답

## 6. REST API 엔드포인트

### 권한 마스터 관리
```http
GET    /api/authorities                  # 모든 권한 조회
GET    /api/authorities/category/{cat}   # 카테고리별 조회
POST   /api/authorities                  # 권한 생성
PUT    /api/authorities/{id}             # 권한 수정
DELETE /api/authorities/{id}             # 권한 삭제
```

### 역할-권한 관리
```http
GET /api/authorities/role/{role}                # 역할의 권한 조회
PUT /api/authorities/role/{role}/mapping        # 역할-권한 매핑 업데이트
```

### 사용자별 개별 권한 관리
```http
GET    /api/authorities/user/{userId}                      # 사용자 개별 권한 조회
GET    /api/authorities/user/{userId}/valid                # 유효한 권한만 조회
POST   /api/authorities/user/{userId}                      # 권한 부여
DELETE /api/authorities/user/{userId}/authority/{authId}   # 특정 권한 회수
DELETE /api/authorities/user/{userId}                      # 모든 개별 권한 회수
```

## 8. 주요 특징 및 장점

### 유연성
- 역할 기반 권한으로 대부분의 사용자 관리
- 개별 권한으로 예외 케이스 처리

### 보안성
- 권한 부여 이력 추적 (granted_by)
- 만료 기능으로 임시 권한 자동 회수
- 중복 권한 방지

### 확장성
- 새로운 권한 추가 용이
- 카테고리로 권한 그룹화
- 권한 계층 구조 확장 가능

### 감사 추적
- 언제, 누가, 어떤 권한을 부여했는지 기록
- 권한 만료 이력 관리
- 권한 부여 사유 기록 (notes)

---

## 9. 데이터 생성 순서

```
1. Authority (권한) 생성
   ↓
2. Role-Authority 매핑 (선택)
   ↓
3. User 생성 with Role
   ↓
4. User-Authority 개별 권한 부여 (선택)
```

**중요:** 권한은 반드시 `authority` 테이블에 먼저 존재해야 하며, 그 후에 역할이나 사용자에게 부여할 수 있습니다.
