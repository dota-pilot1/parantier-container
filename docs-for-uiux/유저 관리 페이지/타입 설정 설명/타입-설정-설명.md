# AG-Grid 타입 설정 설명

## 핵심 질문: 왜 `field`만 UserResponse의 영향을 받나요?

**답변**: AG-Grid 라이브러리 내부에서 `ColDef` 인터페이스가 그렇게 정의되어 있기 때문입니다.

---

## AG-Grid 내부 정의 (우리 눈에 안 보임)

```typescript
// ag-grid-community 라이브러리 내부
interface ColDef<TData = any> {
  field?: keyof TData     // ← TData의 키만 허용
  headerName?: string     // ← 일반 문자열
  width?: number
  sortable?: boolean
  // ...
}
```

**핵심**:
- `field`는 `keyof TData` 타입 → 데이터 타입의 키만 허용
- `headerName`, `width` 등은 일반 타입 → TData와 무관

---

## keyof 연산자

TypeScript의 `keyof`는 객체의 모든 키를 유니온 타입으로 추출합니다.

```typescript
interface UserResponse {
  id: number
  email: string
  username: string
}

type UserKeys = keyof UserResponse
// 결과: 'id' | 'email' | 'username'
```

---

## 실제 동작

```typescript
// 우리가 작성하는 코드
const columnDefs: ColDef<UserResponse>[] = [
  {
    field: 'email',      // ✅ UserResponse의 키
    headerName: '이메일',  // ✅ 아무 문자열
  }
]

// TypeScript가 이해하는 방식
interface ColDef<UserResponse> {
  field?: 'id' | 'email' | 'username' | 'role' | 'isActive' | 'createdAt'
  headerName?: string
}
```

## 왜 이렇게 설계했나?

### `field`는 데이터와 직접 연결

```typescript
const users: UserResponse[] = [
  { id: 1, email: 'test@test.com', username: 'John', ... }
]

const columnDefs = [
  { field: 'email' }  // → users[0].email 값을 가져옴
]
```

**`field: 'invalid'`라면?**
→ `users[0].invalid` = `undefined` → 빈 컬럼

**타입 체크로 방지!**

### `headerName`은 단순 표시 텍스트

```typescript
const columnDefs = [
  {
    field: 'email',
    headerName: '이메일'  // 화면에 표시만 됨 (데이터와 무관)
  }
]
```

## SQL과 비슷함

```sql
SELECT email FROM users           -- field (실제 컬럼명)
SELECT email AS "이메일" FROM users  -- headerName (별칭)
```

## 요약

### Q: 왜 `field`만 UserResponse의 영향을 받나요?

**A**: 라이브러리 내부에서 `field?: keyof TData`로 정의되어 있어서

### Q: `keyof`가 뭔가요?

**A**: 객체의 모든 키를 유니온 타입으로 추출 (`'id' | 'email' | ...`)

### Q: 왜 이렇게 설계했나요?

**A**:
- `field` = 실제 데이터 키 (타입 체크 필요)
- `headerName` = 표시 텍스트 (자유롭게 작성)

**한 줄 요약**: 라이브러리 내부에 `keyof TData`로 이미 정의되어 있어서, 우리 눈에는 안 보이지만 타입 체크가 자동으로 됩니다.

## 10. ColDef 타입이란?

### 정의
`ColDef`는 **Column Definition**의 약자로, AG-Grid 컬럼의 설정을 정의하는 타입입니다.

```typescript
import type { ColDef } from 'ag-grid-community'

const columnDefs: ColDef<UserResponse>[] = [
  {
    headerName: 'ID',
    field: 'id',
    width: 90,
  }
]
```

### 구조 분해

```typescript
ColDef<UserResponse>[]
```

| 부분 | 의미 | 설명 |
|---|---|---|
| `ColDef` | 컬럼 정의 타입 | AG-Grid가 제공하는 타입 |
| `<UserResponse>` | 제네릭 타입 파라미터 | 행 데이터의 타입 |
| `[]` | 배열 | 여러 개의 컬럼 정의 |

## 2. 제네릭 타입의 역할

### 2.1 field 필드 타입 체크

```typescript
interface UserResponse {
  id: number
  email: string
  username: string
  role: 'ROLE_USER' | 'ROLE_ADMIN'
  isActive: boolean
  createdAt: string
}

const columnDefs: ColDef<UserResponse>[] = [
  {
    field: 'email',    // ✅ OK - UserResponse에 존재
    headerName: '이메일',
  },
  {
    field: 'invalid',  // ❌ 타입 에러! UserResponse에 없는 필드
    headerName: '잘못된',
  }
]
```

### 2.2 valueFormatter 타입 추론

```typescript
const columnDefs: ColDef<UserResponse>[] = [
  {
    field: 'email',
    valueFormatter: (params) => {
      // params.value는 string 타입으로 자동 추론됨
      return params.value.toUpperCase()  // ✅ string 메서드 사용 가능
    }
  },
  {
    field: 'id',
    valueFormatter: (params) => {
      // params.value는 number 타입으로 자동 추론됨
      return params.value.toFixed(2)  // ✅ number 메서드 사용 가능
    }
  }
]
```

### 2.3 cellRenderer props 타입 안전성

```typescript
function RoleCellRenderer(props: ICellRendererParams<UserResponse>) {
  // props.data는 UserResponse 타입
  const userId = props.data.id          // ✅ number
  const email = props.data.email        // ✅ string
  const role = props.data.role          // ✅ 'ROLE_USER' | 'ROLE_ADMIN'

  // props.value는 해당 필드의 타입
  const roleValue = props.value         // ✅ 'ROLE_USER' | 'ROLE_ADMIN'

  // 잘못된 필드 접근 시 에러
  const invalid = props.data.invalid    // ❌ 타입 에러!

  return <Select value={roleValue}>...</Select>
}
```

## 3. 주요 AG-Grid 타입들

### 3.1 ColDef (컬럼 정의)

```typescript
import type { ColDef } from 'ag-grid-community'

const columnDefs: ColDef<UserResponse>[] = [
  {
    headerName: 'ID',           // 헤더 이름
    field: 'id',                // 데이터 필드
    width: 90,                  // 컬럼 너비
    sortable: true,             // 정렬 가능 여부
    filter: true,               // 필터 가능 여부
    cellRenderer: MyRenderer,   // 커스텀 렌더러
    cellStyle: {...},           // 셀 스타일
    valueFormatter: (params) => {...},  // 값 포맷팅
  }
]
```

**주요 속성:**
- `field`: 데이터 객체의 필드명 (keyof TData)
- `headerName`: 컬럼 헤더 텍스트
- `width`, `minWidth`, `maxWidth`, `flex`: 너비 설정
- `sortable`, `filter`, `resizable`: 기능 활성화
- `cellRenderer`: 커스텀 셀 렌더러
- `valueFormatter`: 값 포맷팅 함수
- `cellStyle`, `headerClass`: 스타일링

---

### 3.2 ICellRendererParams (셀 렌더러 파라미터)

```typescript
import type { ICellRendererParams } from 'ag-grid-community'

function CustomRenderer(props: ICellRendererParams<UserResponse>) {
  // props 타입 구조
  props.data        // UserResponse - 전체 행 데이터
  props.value       // any - 현재 셀의 값
  props.node        // RowNode - 행 노드 (API 접근용)
  props.colDef      // ColDef - 컬럼 정의
  props.api         // GridApi - Grid API
  props.columnApi   // ColumnApi - 컬럼 API

  return <div>{props.value}</div>
}
```

**주요 속성:**
- `data`: 전체 행 데이터
- `value`: 현재 셀의 값
- `node`: 행 노드 (setDataValue 등 사용)
- `api`: Grid 전역 API
- `colDef`: 해당 컬럼의 정의

---

### 3.3 SelectionChangedEvent (선택 변경 이벤트)

```typescript
import type { SelectionChangedEvent } from 'ag-grid-community'

const handleSelectionChanged = (event: SelectionChangedEvent) => {
  // 선택된 행들 가져오기
  const selectedRows = event.api.getSelectedRows()  // UserResponse[]

  // 선택된 노드들 가져오기
  const selectedNodes = event.api.getSelectedNodes()

  // 전체 행 순회
  event.api.forEachNode((node) => {
    console.log(node.data)  // UserResponse
  })
}
```

**주요 메서드:**
- `event.api.getSelectedRows()`: 선택된 행 데이터 배열
- `event.api.getSelectedNodes()`: 선택된 노드 배열
- `event.api.forEachNode()`: 모든 행 순회

---

### 3.4 AgGridReact (React 컴포넌트)

```typescript
import { AgGridReact } from 'ag-grid-react'

function UsersPage() {
  const gridRef = useRef<AgGridReact>(null)

  return (
    <AgGridReact<UserResponse>  // 제네릭 타입 지정
      ref={gridRef}
      rowData={users}            // UserResponse[]
      columnDefs={columnDefs}    // ColDef<UserResponse>[]
      onSelectionChanged={handleSelectionChanged}
    />
  )
}
```

### 4.1 필드명 오타 감지 불가

```typescript

const columnDefs = [
  {
    field: 'emai',  // 오타인데 에러 안남!
    headerName: '이메일',
  }

// 결과: 런타임에 빈 컬럼으로 표시됨
```

```typescript
// ✅ 타입 있음

  {
    field: 'emai',  // ❌ 컴파일 에러! 'email'을 의미한 건가요?
```

## 8. 타입 체크 흐름

```
1. UserResponse 타입 정의
   ↓
2. ColDef<UserResponse> 사용
   ↓
3. field 필드에 UserResponse의 키만 허용
   ↓
4. ICellRendererParams<UserResponse>에 타입 전달
   ↓
5. cellRenderer 내부에서 props.data가 UserResponse 타입
   ↓
6. 컴파일 타임에 타입 체크
   ↓
7. 런타임 에러 방지
```

## 9. 요약

### ColDef 타입이 필요한 이유

1. **타입 안전성**: 잘못된 필드명 사용 방지
2. **자동완성**: IDE가 사용 가능한 필드/옵션 제안
3. **리팩토링**: 타입 변경 시 관련 코드 자동 감지
4. **문서화**: 코드만 보고도 데이터 구조 파악 가능
5. **런타임 에러 감소**: 컴파일 타임에 대부분의 에러 검출

### 핵심 타입 3가지

```typescript
// 1. ColDef - 컬럼 정의
const columnDefs: ColDef<UserResponse>[] = [...]

// 2. ICellRendererParams - 셀 렌더러 파라미터
function Renderer(props: ICellRendererParams<UserResponse>) {...}

// 3. SelectionChangedEvent - 선택 변경 이벤트
const handleChange = (event: SelectionChangedEvent) => {...}
```

### 타입 사용의 이점

| 타입 없음 | 타입 있음 |
|---|---|
| 런타임 에러 가능 | 컴파일 에러로 사전 방지 |
| 자동완성 없음 | 완벽한 자동완성 |
| 리팩토링 어려움 | 안전한 리팩토링 |
| any 타입 남발 | 정확한 타입 추론 |

**결론:** AG-Grid의 제네릭 타입 시스템은 TypeScript의 강력한 타입 체크를 활용하여 안전하고 유지보수하기 쉬운 코드를 작성할 수 있게 해줍니다!
