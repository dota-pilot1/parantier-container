# 제네릭 타입의 트레이드 오프

## 작성 날짜
2026-03-22

## 핵심 질문

**Q: 제네릭을 사용하면 뭐가 좋고 뭐가 나쁜가?**

**A: 재사용성은 올라가지만 코드 명시성은 떨어진다.**

---

## 1. 트레이드 오프 비교

| 항목 | 제네릭 사용 | 명시적 타입 사용 |
|---|---|---|
| **재사용성** | ⭐⭐⭐⭐⭐ 하나의 인터페이스로 여러 타입 대응 | ⭐ 타입마다 별도 인터페이스 필요 |
| **명시성** | ⭐⭐ `T`가 뭔지 사용처를 봐야 알 수 있음 | ⭐⭐⭐⭐⭐ 타입이 명확하게 보임 |
| **타입 안전성** | ⭐⭐⭐⭐⭐ 컴파일 타임 체크 | ⭐⭐⭐⭐⭐ 컴파일 타임 체크 |
| **유지보수** | ⭐⭐⭐⭐ 한 곳만 수정 | ⭐⭐ 여러 곳 수정 필요 |

**결론: 재사용성 ↑ vs 명시성 ↓의 트레이드 오프**

---

## 2. 예제로 이해하기

### 명시적 타입 (Clarity ⭐⭐⭐⭐⭐, Reusability ⭐)

```typescript
// 각 응답마다 별도 인터페이스
interface UserApiResponse {
  status: number
  message: string
  data: User
}

interface ProductApiResponse {
  status: number
  message: string
  data: Product
}

interface OrderListApiResponse {
  status: number
  message: string
  data: Order[]
}

// ❌ 문제: 같은 구조를 계속 반복
// ✅ 장점: 타입이 명확하게 보임
```

### 제네릭 타입 (Clarity ⭐⭐, Reusability ⭐⭐⭐⭐⭐)

```typescript
// 하나의 제네릭 인터페이스
interface ApiResponse<T> {
  status: number
  message: string
  data: T  // ← T는 나중에 정함
}

// 사용
type UserResponse = ApiResponse<User>
type ProductResponse = ApiResponse<Product>
type OrderListResponse = ApiResponse<Order[]>

// ✅ 장점: 재사용 가능
// ❌ 문제: T가 뭔지 정의를 봐야 알 수 있음
```

## 3. 실제 문제 상황

### 상황 1: 제네릭만 사용 (명시성 부족)

```typescript
interface ApiResponse<T> {
  status: number
  data: T
}

// ❌ 이게 User인지 User[]인지 알 수 없음
function getUsers(): Promise<ApiResponse<???>> {
  // ...
}

// 사용할 때도 헷갈림
const response = await getUsers()
response.data  // ← 이게 단일 객체? 배열?
```

### 상황 2: 명시적 타입만 사용 (중복 코드)

```typescript
interface UserApiResponse {
  status: number
  data: User
}

interface UserListApiResponse {
  status: number
  data: User[]
}

interface ProductApiResponse {
  status: number
  data: Product
}

// ❌ status 구조가 바뀌면?
// 모든 인터페이스를 일일이 수정해야 함
```

## 4. 해결책: 제네릭 + 타입 별칭

**양쪽의 장점을 모두 가져가는 방법**

```typescript
// 1. 제네릭으로 재사용성 확보
interface ApiResponse<T> {
  status: number
  message: string
  data: T
}

// 2. 타입 별칭으로 명시성 확보
type UserResponse = ApiResponse<User>
type UserListResponse = ApiResponse<User[]>
type ProductResponse = ApiResponse<Product>

// 3. 함수 시그니처에서 명시적 타입 사용
function getUser(id: number): Promise<UserResponse> {
  // ✅ 반환 타입이 명확함
}

function getUsers(): Promise<UserListResponse> {
  // ✅ 배열임을 알 수 있음
}

// 4. ApiResponse 구조 변경 시 한 곳만 수정
// 5. 사용처에서는 명확한 타입 이름 사용
```

**결과:**
- ✅ 재사용성: `ApiResponse` 하나만 관리
- ✅ 명시성: `UserResponse`, `UserListResponse` 등 명확한 이름
- ✅ 유지보수: 구조 변경 시 한 곳만 수정

## 5. 언제 무엇을 사용할까?

### 제네릭을 사용하는 경우

```typescript
// ✅ 라이브러리/유틸리티 함수
function fetchData<T>(url: string): Promise<ApiResponse<T>> { ... }

// ✅ 공통 데이터 구조
interface PaginatedResponse<T> {
  items: T[]
  total: number
  page: number
}

// ✅ 상태 관리
interface AsyncState<T> {
  data: T | null
  loading: boolean
  error: Error | null
}
```

### 명시적 타입을 사용하는 경우

```typescript
// ✅ 비즈니스 도메인 모델
interface User {
  id: number
  email: string
  role: UserRole
}

// ✅ 특정 API 응답 (제네릭 + 별칭)
type UserResponse = ApiResponse<User>

// ✅ 단일 사용 인터페이스
interface LoginRequest {
  email: string
  password: string
}
```

## 6. 프로젝트 실제 예시

### AG-Grid의 제네릭 활용

```typescript
// AG-Grid가 제공하는 제네릭
interface ColDef<TData = any> {
  field?: keyof TData
  headerName?: string
  // ...
}

// 프로젝트에서 타입 별칭으로 명시화
const columnDefs: ColDef<UserResponse>[] = [
  {
    field: 'email',     // ✅ UserResponse의 키만 허용
    headerName: '이메일',
  },
]

// ✅ 재사용성: AG-Grid는 모든 타입에 대응
// ✅ 명시성: UserResponse 타입이 명확히 보임
```

### API 응답 타입

```typescript
// 공통 응답 구조 (제네릭)
interface ApiResponse<T> {
  status: number
  message: string
  data: T
}

// 도메인별 타입 별칭 (명시적)
type UserResponse = ApiResponse<User>
type UserListResponse = ApiResponse<User[]>

// API 함수에서 사용
export const adminApi = {
  getUsers: (): Promise<UserListResponse> => {
    return api.get('/admin/users').then(res => res.data)
  },

  getUser: (id: number): Promise<UserResponse> => {
    return api.get(`/admin/users/${id}`).then(res => res.data)
  },
}

// ✅ 함수 시그니처만 봐도 반환 타입이 명확
```

## 7. 실전 가이드라인

### DO: 제네릭 + 타입 별칭 조합

```typescript
// ✅ GOOD: 재사용성 + 명시성
interface Container<T> {
  value: T
}

type UserContainer = Container<User>
type ProductContainer = Container<Product>

function processUser(container: UserContainer) {
  // 타입이 명확함
}
```

### DON'T: 제네릭만 남발

```typescript
// ❌ BAD: 명시성 부족
function process<T>(data: T): Container<T> {
  // T가 뭔지 알 수 없음
}

// 사용처
const result = process(user)  // Container<????>
```

### DO: 적절한 명명

```typescript
// ✅ GOOD: 의미 있는 타입 별칭
type PaginatedUsers = PaginatedResponse<User>
type AsyncUserState = AsyncState<User>

// ❌ BAD: 의미 없는 이름
type T1 = ApiResponse<User>
type T2 = ApiResponse<Product>
```

## 8. 핵심 요약

```typescript
// 패턴 공식

// 1. 제네릭으로 재사용 가능한 구조 정의
interface ApiResponse<T> {
  status: number
  data: T
}

// 2. 타입 별칭으로 구체적인 타입 명시
type UserResponse = ApiResponse<User>

// 3. 함수 시그니처에서 구체적 타입 사용
function getUser(): Promise<UserResponse> { }

// 결과: 재사용성 ⭐⭐⭐⭐⭐ + 명시성 ⭐⭐⭐⭐⭐
```

**핵심 원칙:**
1. **라이브러리/유틸리티 레벨**: 제네릭 사용
2. **애플리케이션 레벨**: 타입 별칭으로 구체화
3. **함수 시그니처**: 명시적 타입 사용

이렇게 하면 "마법 같은" 제네릭의 재사용성을 유지하면서도, 코드 명시성을 해치지 않을 수 있습니다!
