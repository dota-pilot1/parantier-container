# 메뉴 시스템 아키텍처

## 설계 원칙

우리 프로젝트의 메뉴 시스템은 **하이브리드 접근 방식**을 채택합니다:

- **헤더 메뉴 (Header Menu)**: DB 관리 - 비즈니스 요구사항에 따라 유연하게 변경
- **사이드바 메뉴 (Sidebar Menu)**: Config 파일 관리 - 시스템 구조, 명시적이고 타입 안전

---

## 왜 이런 구조인가?

### 문제 인식

초기에는 모든 메뉴를 DB로 관리했으나, 다음과 같은 문제점이 있었습니다:

1. **비명시적 로직**: 권한 체크가 어디서 일어나는지 불명확
2. **복잡성 증가**: DB → Backend → API → Frontend → Parse → Render (6단계)
3. **유지보수 어려움**: 메뉴 구조 변경 시 여러 곳을 수정
4. **타입 안정성 부족**: 런타임에만 오류 발견
5. **추적 불가능**: Git으로 변경사항 추적 불가

### 해결책

**책임의 분리 (Separation of Concerns)**

```
┌─────────────────────────────────────────────────┐
│                   Header Menu                    │
│              (비즈니스 영역 - DB 관리)             │
│  - 고객사마다 다를 수 있는 메뉴                     │
│  - 예: "영업 관리", "재고 관리", "회계 관리"         │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│                  Sidebar Menu                    │
│             (시스템 영역 - Config 관리)            │
│  - 시스템 설정, 관리자 기능                         │
│  - 예: "유저 관리", "권한 관리", "조직 관리"         │
└─────────────────────────────────────────────────┘
```

---

## 구현 상세

### 1. Sidebar Config 방식 (교과서적 구현)

**파일**: `parantier-front/src/config/sidebar.ts`

```typescript
export const sidebarConfig: Record<string, SidebarSection[]> = {
  admin: [
    {
      title: '사용자 관리',
      items: [
        {
          label: '유저 관리',
          path: '/admin/users',
          icon: Users,
          authority: 'USER:READ',  // ← 권한이 코드에 명시적으로 보임
        },
        {
          label: '조직 관리',
          path: '/admin/organizations',
          icon: Building2,
          authority: 'ORGANIZATION:READ',
        },
      ],
    },
    {
      title: '시스템 관리',
      items: [
        {
          label: '권한 관리',
          path: '/admin/authorities',
          icon: Shield,
          authority: 'AUTHORITY:READ',
        },
      ],
    },
  ],
}
```

**장점**:
- ✅ **명시적**: 모든 메뉴와 권한이 한눈에 보임
- ✅ **타입 안전**: TypeScript로 컴파일 타임에 오류 검증
- ✅ **Git 추적**: 변경사항이 버전 관리됨
- ✅ **핫 리로드**: 코드 수정 시 즉시 반영
- ✅ **테스트 가능**: 단위 테스트 작성 가능
- ✅ **IDE 지원**: 자동완성, 타입 검사, 리팩토링 지원

**단점**:
- ❌ 메뉴 순서 변경 시 코드 수정 필요
- ❌ 비개발자가 직접 수정 불가

---

### 2. Header Menu DB 방식

**테이블**: `menus`

```sql
CREATE TABLE menus (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    path VARCHAR(255),
    parent_id BIGINT REFERENCES menus(id),
    menu_type VARCHAR(20) NOT NULL,  -- HEADER, CATEGORY, PAGE
    order_num INTEGER DEFAULT 0,
    icon VARCHAR(50),
    is_active BOOLEAN DEFAULT true
);
```

**장점**:
- ✅ **유연성**: 런타임에 메뉴 구조 변경 가능
- ✅ **비즈니스 요구사항**: 고객사별 커스터마이징 가능
- ✅ **1-3차 계층 구조**: 복잡한 메뉴 트리 지원

**단점**:
- ❌ 권한 체크 로직이 숨겨짐
- ❌ 런타임 오류 가능
- ❌ Git 추적 불가
- ❌ 타입 안정성 낮음

### Sidebar 메뉴 추가 (권장)

**시나리오**: 새로운 관리자 기능 "로그 관리" 추가

1. **sidebar.ts 수정**:

```typescript
import { FileText } from 'lucide-react'

export const sidebarConfig = {
  admin: [
    {
      title: '시스템 관리',
      items: [
        // ... 기존 메뉴들
        {
          label: '로그 관리',
          path: '/admin/logs',
          icon: FileText,
          authority: 'LOG:READ',
        },
      ],
    },
  ],
}
```

2. **권한 추가** (DB):
```sql
INSERT INTO authority (name, description, category)
VALUES ('LOG:READ', '로그 조회 권한', 'LOG');

INSERT INTO role_authority (role, authority_id)
SELECT 'ROLE_ADMIN', id FROM authority WHERE name = 'LOG:READ';
```

3. **라우트 추가** (routes/index.tsx):
```typescript
const adminLogsRoute = createRoute({
  getParentRoute: () => rootRoute,
  path: '/admin/logs',
  beforeLoad: () => requireAuthority('LOG:READ'),
  component: LogsPage,
})
```

**끝!** 3단계로 완료.

## 권한 체크 흐름

### Sidebar (Config 방식)

```typescript
// 1. Config에서 메뉴 가져오기
const sections = sidebarConfig['admin']

// 2. 사용자 권한으로 필터링
const userAuthorities = ['USER:READ', 'ORGANIZATION:READ']
const filteredSections = filterSidebarByAuthority(sections, userAuthorities)

// 3. 렌더링
{filteredSections.map(section => <SidebarSection {...section} />)}
```

**단계**: Config → Filter → Render (2단계)

---

### Header (DB 방식)

```typescript
// 1. DB에서 메뉴 fetch
const menus = await menuApi.getTree()

// 2. 백엔드에서 권한 체크 (선택)
// 3. 프론트에서 파싱
const headerMenus = menus.filter(menu => menu.menuType === 'HEADER')

// 4. 렌더링
{headerMenus.map(menu => <HeaderItem {...menu} />)}
```

**단계**: DB → Backend → API → Frontend → Parse → Render (6단계)

## 성능 고려사항

### Sidebar
- ✅ **정적 import**: 번들에 포함, 추가 네트워크 요청 없음
- ✅ **Tree shaking**: 사용하지 않는 아이콘 제거
- ✅ **타입 체크**: 빌드 타임에 검증

### Header
- ⚠️ **Runtime fetch**: 페이지 로드 시 API 호출 필요
- ⚠️ **캐싱 필요**: React Query로 캐싱 구현
- ⚠️ **로딩 상태**: Skeleton UI 필요

## 마이그레이션 가이드

기존 DB 메뉴를 Config로 전환하려면:

1. **현재 메뉴 구조 확인**:
```sql
SELECT * FROM menus WHERE menu_type = 'CATEGORY';
```

2. **sidebar.ts로 이동**:
```typescript
export const sidebarConfig = {
  [categoryKey]: [
    {
      title: 'Section Name',
      items: [/* DB 메뉴를 여기로 복사 */]
    }
  ]
}
```

3. **DB 메뉴 삭제** (선택):
```sql
UPDATE menus SET is_active = false WHERE menu_type = 'CATEGORY';
```

## 결론

### Config 방식 사용 시점
- ✅ 시스템/관리자 메뉴
- ✅ 권한 기반 액세스 제어
- ✅ 자주 바뀌지 않는 구조
- ✅ 타입 안정성이 중요한 경우

### DB 방식 사용 시점
- ✅ 비즈니스 도메인 메뉴
- ✅ 고객사별 커스터마이징
- ✅ 런타임 동적 변경 필요
- ✅ 1-3차 복잡한 계층 구조

**핵심**: 두 방식을 혼용하되, **시스템 메뉴는 Config, 비즈니스 메뉴는 DB**로 관리하는 것이 균형 잡힌 접근입니다.
