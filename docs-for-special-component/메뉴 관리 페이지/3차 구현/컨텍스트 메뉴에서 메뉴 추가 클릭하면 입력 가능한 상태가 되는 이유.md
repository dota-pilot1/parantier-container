# 컨텍스트 메뉴에서 "하위 메뉴 추가" 클릭 시 입력 가능한 상태가 되는 이유

## 날짜
2026-03-23

## 개요
우클릭 컨텍스트 메뉴에서 "하위 메뉴 추가"를 클릭하면 트리에 인라인 입력 필드가 나타나고 자동으로 포커스되는 메커니즘을 설명합니다.

## 전체 흐름

```
사용자 우클릭
  ↓
컨텍스트 메뉴 표시
  ↓
"하위 메뉴 추가" 클릭
  ↓
상태 변경: setAddingChildToId(부모메뉴.id)
  ↓
TreeNode 리렌더링
  ↓
조건 충족: isExpanded && addingChildToId === menu.id
  ↓
InlineMenuInput 컴포넌트 마운트
  ↓
useEffect 실행: inputRef.current.focus()
  ↓
입력 필드 자동 포커스
```

## 1단계: 우클릭 이벤트 캡처

### TreeNode.tsx
```tsx
const handleContextMenu = (e: React.MouseEvent) => {
  e.preventDefault()         // 브라우저 기본 컨텍스트 메뉴 방지
  e.stopPropagation()        // 이벤트 버블링 중단
  onContextMenu(e.clientX, e.clientY, menu)  // 부모에게 전달
}

<div
  onClick={() => onSelect(menu)}
  onContextMenu={handleContextMenu}
>
```

**중요 포인트:**
- `e.preventDefault()`: 브라우저의 기본 우클릭 메뉴가 나타나지 않도록 방지
- `e.stopPropagation()`: 부모 요소로 이벤트가 전파되지 않도록 차단
- `e.clientX, e.clientY`: 마우스 클릭 위치 좌표를 전달하여 컨텍스트 메뉴를 해당 위치에 표시

## 2단계: 컨텍스트 메뉴 표시

### MenusPage.tsx
```tsx
const [contextMenu, setContextMenu] = useState<{
  x: number
  y: number
  menu: Menu
} | null>(null)

const handleContextMenu = (x: number, y: number, menu: Menu) => {
  setContextMenu({ x, y, menu })
}
```

**상태 구조:**
- `x, y`: 컨텍스트 메뉴를 표시할 화면 좌표
- `menu`: 우클릭된 메뉴 객체 (부모가 될 메뉴)
- `null`: 컨텍스트 메뉴가 닫혀있는 상태

### ContextMenu.tsx 렌더링
```tsx
{contextMenu && (
  <ContextMenu
    x={contextMenu.x}
    y={contextMenu.y}
    onClose={closeContextMenu}
    onAddChild={handleAddChildMenu}
    onEdit={handleEditFromContext}
    onDelete={handleDeleteFromContext}
  />
)}
```

**렌더링 조건:**
- `contextMenu`가 `null`이 아니면 컴포넌트 표시
- `fixed` position으로 `x, y` 좌표에 정확히 배치

## 3단계: "하위 메뉴 추가" 클릭

### ContextMenu.tsx
```tsx
<button
  className="w-full px-4 py-2 text-left text-sm hover:bg-gray-100"
  onClick={() => {
    onAddChild()    // ← 핵심: 부모 컴포넌트의 handleAddChildMenu 호출
    onClose()       // 컨텍스트 메뉴 닫기
  }}
>
  ➕ 하위 메뉴 추가
</button>
```

## 4단계: 인라인 입력 모드 활성화

### MenusPage.tsx - handleAddChildMenu()
```tsx
const handleAddChildMenu = () => {
  if (contextMenu) {
    // 1. 부모 메뉴 자동 확장
    setExpandedIds((prev) => new Set(prev).add(contextMenu.menu.id))
    
    // 2. 인라인 입력 모드 활성화 (핵심!)
    setAddingChildToId(contextMenu.menu.id)
  }
}
```

**두 가지 상태 변경:**

1. **`setExpandedIds`**: 부모 메뉴가 접혀있으면 펼치기
   - 자식 입력 필드가 보이려면 부모가 펼쳐져 있어야 함
   - `new Set(prev).add(id)`: 기존 확장 상태 유지 + 새로운 ID 추가

2. **`setAddingChildToId`**: 어떤 메뉴에 자식을 추가 중인지 표시
   - `null` → `number`: 인라인 입력 활성화
   - 이 값이 TreeNode의 렌더링 조건으로 사용됨

## 5단계: TreeNode 리렌더링 및 조건부 렌더링

### TreeNode.tsx
```tsx
{/* 자식 노드들 먼저 렌더링 */}
{hasChildren && isExpanded && (
  <div>
    {children.map((child) => <TreeNode ... />)}
  </div>
)}

{/* 인라인 입력 (하위 메뉴 추가) */}
{isExpanded && addingChildToId === menu.id && (
  <InlineMenuInput
    depth={depth + 1}
    onSubmit={(name) => onInlineSubmit(menu.id, name)}
    onCancel={onInlineCancel}
  />
)}
```

**렌더링 조건 분석:**

| 조건 | 의미 | 이유 |
|------|------|------|
| `isExpanded` | 부모 메뉴가 펼쳐져 있음 | 접혀있으면 자식 입력 필드가 안 보여야 함 |
| `addingChildToId === menu.id` | 현재 메뉴에 자식 추가 중 | 다른 메뉴에는 입력 필드가 나타나면 안 됨 |

**들여쓰기:**
- `depth={depth + 1}`: 부모보다 한 단계 더 들여쓰기
- 시각적으로 부모-자식 관계 명확히 표현

## 6단계: InlineMenuInput 마운트 및 자동 포커스

### InlineMenuInput.tsx
```tsx
export function InlineMenuInput({ depth, onSubmit, onCancel }: InlineMenuInputProps) {
  const inputRef = useRef<HTMLInputElement>(null)

  // 컴포넌트 마운트 시 자동 포커스
  useEffect(() => {
    inputRef.current?.focus()
  }, [])

  return (
    <div ...>
      <input
        ref={inputRef}
        type="text"
        ...
      />
    </div>
  )
}
```

**자동 포커스 메커니즘:**

1. **`useRef<HTMLInputElement>(null)`**
   - DOM 요소에 직접 접근하기 위한 ref 생성

2. **`ref={inputRef}`**
   - input 요소와 ref 연결

3. **`useEffect(() => { inputRef.current?.focus() }, [])`**
   - 빈 의존성 배열 `[]`: 컴포넌트 마운트 시 **딱 한 번만** 실행
   - `inputRef.current?.focus()`: input 요소에 포커스 설정
   - `?.` 옵셔널 체이닝: ref가 아직 연결되지 않았을 경우 에러 방지

**결과:**
- 인라인 입력 필드가 나타나자마자 자동으로 포커스됨
- 사용자가 추가 클릭 없이 바로 타이핑 시작 가능

## 상태 흐름 다이어그램

```
[초기 상태]
addingChildToId = null
→ 인라인 입력 필드 없음

    ↓ 우클릭 + "하위 메뉴 추가" 클릭

[인라인 입력 활성]
addingChildToId = 10 (유저 관리의 ID)
expandedIds = Set([10])
→ "유저 관리" 아래 입력 필드 표시
→ 자동 포커스

    ↓ Enter 키 또는 Esc 키

[인라인 입력 비활성]
addingChildToId = null
→ 입력 필드 사라짐
```

## 왜 이렇게 구현했는가?

### 1. 상태 기반 렌더링 (State-Driven Rendering)
```tsx
// ❌ 나쁜 예: DOM 직접 조작
document.getElementById('inline-input').style.display = 'block'

// ✅ 좋은 예: 상태로 렌더링 제어
{addingChildToId === menu.id && <InlineMenuInput />}
```

**장점:**
- React의 선언적 패러다임에 부합
- 상태와 UI가 항상 동기화됨
- 디버깅이 쉬움 (React DevTools로 상태 추적 가능)

### 2. 단일 상태로 전역 제어
```tsx
const [addingChildToId, setAddingChildToId] = useState<number | null>(null)
```

**장점:**
- 한 번에 하나의 메뉴에만 입력 필드가 나타남
- 여러 메뉴에 동시에 입력 필드가 생기는 것을 방지
- 상태 관리가 단순함

### 3. Props Drilling을 통한 이벤트 전파
```
ContextMenu
  → MenusPage.handleAddChildMenu()
  → setAddingChildToId(id)
  → TreeView (props 전달)
  → TreeNode (props 전달)
  → 조건부 렌더링
```

**왜 Context API를 사용하지 않았나?**
- 트리 구조가 깊지 않음 (2-3단계)
- Props drilling이 오히려 명확하고 추적하기 쉬움
- Context API는 과도한 추상화가 될 수 있음

## 자동 포커스의 중요성

### useEffect를 사용하는 이유
```tsx
// ❌ 작동하지 않음: 렌더링 시점에는 DOM이 아직 생성되지 않음
const InlineMenuInput = () => {
  const inputRef = useRef<HTMLInputElement>(null)
  inputRef.current?.focus()  // null!
  return <input ref={inputRef} />
}

// ✅ 작동함: DOM 생성 후 실행
const InlineMenuInput = () => {
  const inputRef = useRef<HTMLInputElement>(null)
  useEffect(() => {
    inputRef.current?.focus()  // DOM 준비됨!
  }, [])
  return <input ref={inputRef} />
}
```

**React 라이프사이클:**
1. 함수 컴포넌트 실행 (JSX 반환)
2. Virtual DOM 비교
3. Real DOM 업데이트
4. **useEffect 실행** ← 이 시점에 DOM이 준비됨

## 결론

컨텍스트 메뉴에서 인라인 입력이 나타나는 메커니즘은:

1. **이벤트 캡처**: 우클릭 → 좌표와 메뉴 정보 저장
2. **상태 변경**: `addingChildToId` 설정
3. **조건부 렌더링**: `addingChildToId === menu.id`일 때만 표시
4. **자동 포커스**: `useEffect` + `ref.focus()`로 즉시 입력 가능

이 모든 과정이 React의 **선언적 렌더링**과 **단방향 데이터 흐름** 원칙에 따라 작동합니다.
